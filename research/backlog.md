# BPS-Engine 待优化清单

> 来源标注：[帖子分析] = 来自外部社区帖子的借鉴启发；[内部] = 项目内部发现

---

## 1. 显式并行语义 [帖子分析]

**现状**：composite service 可通过 `call_sub_service` 调用多个子服务，但蓝图 YAML 中缺乏显式的并行/串行声明。

**优化方向**：在蓝图 YAML 中增加 `parallel` / `join` 原语，让蓝图设计者能显式声明哪些子服务可并行执行。类比 OS 的 fork/join 语义。

**启发来源**：Opus-4.6 观点——"编排应像 OS 进程管理，分解依据是计算效率（并行性）"。

---

## 2. Context-aware 服务粒度拆分 [帖子分析]

**现状**：Service 拆分粒度主要基于业务语义，未考虑 LLM 时代特有的计算约束。

**优化方向**：Service 拆分指导原则中应加入"单个 Agent context window 能否承载"这一约束。context window 之于 Agent，类比 OS 中的内存限制之于进程——是硬性的物理约束，应影响服务拆分决策。

**启发来源**：Opus-4.6 观点——"分解依据包括上下文窗口"。

---

## 3. 进程组级错误传播控制 [帖子分析]

**现状**：Process 有 ERROR 状态和 EventEmitter 事件机制，但缺少层级化的错误传播控制。一个子服务出错时，对兄弟子服务和父服务的影响策略不够灵活。

**优化方向**：引入类似 OS process group / cgroup 的概念，支持：
- 错误隔离（子服务失败不影响兄弟）
- 错误传播（子服务失败导致整个组取消）
- 自定义策略（蓝图中声明错误处理策略）

**启发来源**：Opus-4.6 观点——"分解依据包括错误隔离"。
