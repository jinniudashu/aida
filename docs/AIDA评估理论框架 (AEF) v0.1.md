# AIDA 评估理论框架 (AEF) v0.1

> AIDA Evaluation Framework — 基于四元理论的可复用工程进展观测方法论
>
> 目的：给定阶段测评结果，系统性产出（1）当前能力结构维度下需改进的问题（2）新增结构能力建议

---

## 一、四元理论基座

每一元理论对 AIDA 提出不同的诊断问题。单一理论有盲区，四元交叉验证消除偏见。

### 1.1 操作系统理论（Dijkstra, Thompson）

**诊断问题**：机制是否**正确实现**？

将 AIDA 映射为 Agent 运行时操作系统：
- Entity = 内存页（版本化，全局寻址）
- Task/Process = 进程（5-state 状态机）
- Management Layer = 保护环（Ring 0 ActionGate vs Ring 3 Agent）
- bps_scan_work = 调度器（优先级队列）
- Blueprint = 可执行程序（编译后加载）
- EventEmitter = IPC（广播式）

**典型诊断输出**：状态不可达、资源泄漏、权限绕过、调度饥饿。

### 1.2 控制论（Wiener, Ashby）

**诊断问题**：机制是否**有效调节**？

将 AIDA 映射为反馈控制系统：
- Management 约束 = 控制器
- Agent 行为 = 被控对象
- Entity 状态 = 被控变量
- 熔断器 = 唯一全自动反馈回路
- 审批 = 人在回路（半自动）

**核心定律**：
- **Ashby 必要多样性定律**：控制器多样性必须 ≥ 被控系统多样性。当前约束数（~50）<< 写操作上下文数（~10⁴），安全足够但优化不足。
- **负反馈 → 稳定，正反馈 → 增长/失控**：熔断器是负反馈，Skill 自进化是正反馈，需要平衡。

**典型诊断输出**：开环比例过高、反馈延迟、振荡、控制器失配。

### 1.3 有限理性（Simon）

**诊断问题**：机制是否**对 Agent 可用**？

将 AIDA 映射为有限理性决策环境：
- Agent (LLM) = 有限理性决策者（有限上下文、易错、注意力有限）
- BPS 工具集 = 决策支持基础设施
- Workspace = 决策架构（信息呈现方式影响决策质量）

**核心概念**：
- **满意化（Satisficing）**：Agent 不需要最优解，需要"足够好，现在执行"的信号。
- **注意力分配**：信息过载 → 决策瘫痪（"说而不做"反模式的根因）。
- **搜索-执行鸿沟**：没有信号告知"信息已充分，应执行"。

**典型诊断输出**：信息过载、决策瘫痪、搜索-执行脱节、满意化阈值缺失。

### 1.4 认知导航-重映射理论（嵌入空间框架）

**诊断问题**：机制是否同时支持**利用（导航）与探索（重映射）**？

将 AIDA 映射为嵌入空间中的认知系统：
- BPS 六元组 = 嵌入空间构造（高维组织运营 → 6 维形式化表征）
- 确定性规则执行 = 已知空间中的导航（误差为零）
- 非确定性 LLM 判断 = 遇到新情境时的重映射
- 管理层熔断器 = 近临界调节器（稳定 ↔ 适应的动态平衡）
- HITL 审批 = 协同重映射（非单向控制）
- 三层架构 = 嵌套嵌入空间（高层塑造低层约束景观）

**核心概念**：
- **导航效率**：在已知空间中目标导向行动的效率（工具调用 / 产出比）。
- **重映射能力**：空间结构不足时创造新维度的能力（Skill 创建、Blueprint 进化）。
- **近临界性**：最优工作点不是固定阈值，而是对扰动最敏感的动态状态。
- **协同重映射**：alignment 不是单向控制，是共享问题空间的双向调整。
- **层级嵌套**：高层通过塑造约束景观间接控制低层，而非直接指令。

**典型诊断输出**：导航-重映射失耦（"说而不做"）、近临界偏移（过刚/过柔）、层级穿透（越级操作）、重映射停滞（无法学习新模式）。

### Σ1 过程生命周期（PROC）

**定义**：Task/Process 从创建到完成的全生命周期管理能力。

**理论锚点**：
- OS：进程状态机完备性（所有状态可达、不可达状态无死锁）
- 控制论：进程作为被控对象，状态转移受规则约束
- 认知：进程执行 = 在已知空间中的导航

**健康指标**：
- 5-state 状态机所有转移路径可达（OPEN→IN_PROGRESS→COMPLETED/FAILED/BLOCKED）
- groupId 支持批量操作
- priority + deadline 暴露给调度器
- outcome 结构化（success/partial/failed）

**病征模式**：
| 测试症状 | 理论解释 | 维度内改进方向 |
|----------|----------|---------------|
| 任务永远停在 IN_PROGRESS | OS: 缺少超时/看门狗 | 添加 task timeout 或 heartbeat 检测 |
| bps_batch_update 无效 | OS: 进程组信号缺失 | 检查 groupId 索引和 filterState 逻辑 |
| outcome 全部 success | 有限理性: 满意化阈值缺失 | Agent 需要 partial/failed 判断指引 |

**当前检查点**：S2.05-S2.12, D4（进程组 4 项）

### Σ2 实体架构（ENTITY）

**定义**：业务对象（Dossier）的存储、版本化、关系管理能力。

**理论锚点**：
- OS：内存管理（版本化 = CoW 快照，erpsysId = 全局寻址）
- 认知：实体结构 = 嵌入空间的骨架（维度定义了问题空间的基础坐标）

**健康指标**：
- 版本递增、smartMerge 数组追加语义
- 关系声明（depends_on/part_of/references）+ relatedEntities 解析
- brief 模式查询（降低 Agent 认知负载）

**病征模式**：
| 测试症状 | 理论解释 | 维度内改进方向 |
|----------|----------|---------------|
| 实体数据丢失（数组覆盖） | OS: 写时复制缺陷 | smartMerge 逻辑校验 |
| relatedEntities 为空 | 认知: 嵌入空间维度间无连接 | 关系声明完整性检查 |
| 实体查询返回过量数据 | 有限理性: 信息过载 | brief 模式启用率统计 |

**当前检查点**：S2.01-S2.04, D5（实体关系 5 项）

### Σ3 约束执行（GATE）

**定义**：管理层对写操作的前置拦截能力——覆盖率、精度、scope 匹配。

**理论锚点**：
- OS：保护环（Ring 0 拦截 Ring 3 的违规操作）
- 控制论：约束 = 控制器，verdict = 控制输出（PASS/BLOCK/REQUIRE_APPROVAL）

**健康指标**：
- GATED_WRITE_TOOLS 覆盖全部 9 个写操作
- scope 匹配精度（entityType + tools 过滤正确）
- undefined variable → PASS（约束不适用于此操作）
- 语法错误 → BLOCK（fail-closed）

**病征模式**：
| 测试症状 | 理论解释 | 维度内改进方向 |
|----------|----------|---------------|
| 管理绕过（write 工具直接写文件） | OS: 保护环被绕过 | exec allowlist + 工具覆盖率检查 |
| 过度拦截（所有操作被 BLOCK） | 控制论: 控制器失配 | scope.dataFields 精确化 |
| 约束从未触发 | 控制论: 控制器死区 | 约束条件与实际操作上下文对齐检查 |

**当前检查点**：D1（管理控制 10 项）, D8（工具注册 2 项）

### Σ4 稳定性调节（STAB）

**定义**：系统在异常累积时的自动保护与恢复能力。

**理论锚点**：
- 控制论：负反馈实现稳态（熔断器 = 唯一全自动反馈回路）
- 认知：近临界性——最优工作点在稳定与适应之间动态滑动

**健康指标**：
- 熔断器 4-state 升级路径正确（NORMAL→WARNING→RESTRICTED→DISCONNECTED）
- 冷却恢复降级路径正确（反向）
- 振荡检测（1h 内 >3 次转移 → 锁定）
- 稳定态下对新信息仍敏感（不过度保守）

**病征模式**：
| 测试症状 | 理论解释 | 维度内改进方向 |
|----------|----------|---------------|
| 熔断器一路升级不回落 | 控制论: 只有负反馈无恢复 | 冷却期参数调优 |
| 频繁 NORMAL↔WARNING 切换 | 控制论: 振荡（调节增益过高） | 振荡检测阈值校准 |
| 系统过度保守（全部 RESTRICTED） | 认知: 偏离近临界——过于刚性 | 降低升级灵敏度或引入分级约束 |
| 系统过度宽松（从不触发） | 认知: 偏离近临界——过于柔性 | 约束条件强化或自动建议新约束 |

**当前检查点**：D2（熔断器 6 项）

---

### Σ5 反馈学习（FDBK）

**定义**：系统从历史运行数据中学习并改进自身规则的能力。

**理论锚点**：
- 控制论：闭环反馈——输出影响未来输入（违规统计 → 约束调整建议）
- 认知：预测-校正循环——每次运行产生的误差信号驱动系统更新

**健康指标**：
- constraintEffectiveness 数据完整（violationCount + approvalRate）
- 策略建议准确（通过率 >90% → "考虑放宽"；拒绝率 >80% → "考虑升级"）
- Skill 使用追踪（dormantSkills 检测 90 天未用技能）
- outcomeDistribution 提供任务完成质量分布

**病征模式**：
| 测试症状 | 理论解释 | 维度内改进方向 |
|----------|----------|---------------|
| constraintEffectiveness 全部无建议 | 控制论: 样本不足无法统计 | 降低样本阈值或累积多轮数据 |
| 所有约束建议"放宽" | 控制论: 控制器过严 | 执行放宽建议，调整阈值 |
| dormantSkills 累积但无人处理 | 认知: 重映射产物未维护 | 添加 dormant Skill 清理提醒 |

**当前检查点**：D7（约束分析 3 项）, D6（Skill 指标 3 项）

### Σ6 信息呈现（INFO）

**定义**：系统向 Agent 呈现信息的质量——Agent 能否从返回数据中快速做出正确决策。

**理论锚点**：
- 有限理性：信息呈现方式决定决策质量（决策架构效应）
- 认知：梯度信号传播——Agent 需要清晰的误差信号才能有效导航

**健康指标**：
- bps_scan_work 返回 top-5 + `{items, total, showing}` 元数据 + summary 一行摘要
- bps_query_entities 支持 brief 模式
- bps_next_steps 返回 recommendation
- sortByUrgency 按 deadline ASC, priority DESC 排序

**病征模式**：
| 测试症状 | 理论解释 | 维度内改进方向 |
|----------|----------|---------------|
| Agent 反复调用 query 但不行动 | 有限理性: 搜索-执行鸿沟 | 添加"信息充分度"信号（"3 entities match, no more results"） |
| Agent 忽略 recommendation | 认知: 梯度信号太弱 | 提高 recommendation 的可操作性（从建议→具体指令） |
| Agent 请求全量数据而非 brief | 有限理性: 默认行为偏向全量 | 工具默认返回 brief，需显式请求 full |

**当前检查点**：D3（信息摘要 6 项）

### Σ7 调度效率（SCHED）

**定义**：系统帮助 Agent 选择"下一步做什么"的能力——优先级排序、截止日期感知、紧急度计算。

**理论锚点**：
- OS：调度器（priority queue + deadline scheduling）
- 有限理性：注意力分配——在有限时间内做最重要的事
- 认知：导航效率——在问题空间中选择最短路径而非随机游走

**健康指标**：
- priority + deadline 字段被 Agent 使用（非全部默认值）
- overdueTasks 正确标记逾期任务
- sortByUrgency 排序结果与直觉一致
- Agent 实际按排序顺序处理任务

**病征模式**：
| 测试症状 | 理论解释 | 维度内改进方向 |
|----------|----------|---------------|
| 所有任务 priority=0 | 有限理性: 决策架构未强制优先级 | bps_create_task 引导或默认规则 |
| Agent 先做低优先级任务 | 认知: 导航效率低 | bps_scan_work 置顶高优任务 |
| deadline 过期无响应 | OS: 无看门狗 | 逾期任务自动提醒或状态标记 |

**当前检查点**：S2.09-S2.12（调度相关）

### Σ8 自我演进（EVOL）

**定义**：系统在运行时创造新能力（Skill、Agent、Cron）并维护已有能力的能力。

**理论锚点**：
- 控制论：超稳定性（Ashby）——系统在一级反馈失败时切换到二级参数调整
- 认知：重映射能力——当现有空间不足以解决问题时，创造新的维度

**健康指标**：
- bps_create_skill 成功率
- bps_register_agent 成功率（不被管理过度拦截）
- Cron 任务注册 + Gateway 重启后存活
- dormantSkills 检测 + 清理
- Skill 使用指标（invokedAt, outcome, durationMs）

**病征模式**：
| 测试症状 | 理论解释 | 维度内改进方向 |
|----------|----------|---------------|
| 0 个 Skill 创建 | 认知: 重映射能力缺失 | AGENTS.md 自进化指引强化 |
| Skill 创建被管理拦截 | 控制论: 控制器过严抑制适应 | 约束 scope 排除 bps_create_skill 或专用审批通道 |
| Cron 全部丢失 | OS: 无持久化 | BOOT.md cron 恢复步骤验证 |
| 大量 dormant Skills | 认知: 重映射过度（创建 > 使用） | Skill 合并或清理机制 |

**当前检查点**：D6（Skill 指标 3 项）, V5（验证 Skill/Agent/Cron 创建）

---

### Σ9 层级一致性（HIER）

**定义**：管理层（Blueprint/约束）与运营层（Entity/Skill）之间的正确分层——高层塑造约束景观，低层在景观中自由导航。

**理论锚点**：
- OS：特权环（Ring 0 不应执行 Ring 3 业务逻辑，Ring 3 不应绕过 Ring 0 保护）
- 控制论：层级控制——高层设定参数范围，低层在范围内自主优化
- 认知：嵌套嵌入空间——高层塑造低层能量景观

**健康指标**：
- 二层路由准确率 100%（约束 → 管理层, 内容/探测 → 运营层）
- Blueprint 仅包含约束/审批门/禁令，不包含运营细节
- Entity + Skill 作为运营主路径
- Agent 不越级修改管理层配置（除非经审批）

**病征模式**：
| 测试症状 | 理论解释 | 维度内改进方向 |
|----------|----------|---------------|
| 运营内容被写入 Blueprint | 认知: 层级穿透——低层信息泄漏到高层 | blueprint-modeling Skill scope 强化 |
| 管理约束不经审批被修改 | OS: Ring 3 写入 Ring 0 | bps_load_management 管理门控 |
| Agent 同时用 Blueprint 和 Entity 做同一件事 | 认知: 嵌入空间冗余 | action-plan Skill 层分类强制执行 |

**当前检查点**：B4 二层路由相关检查

### Σ10 协同适应（COADAPT）

**定义**：人类与 Agent 通过管理闭环（审批/拒绝/效能反馈）共同调整共享操作空间的能力。

**理论锚点**：
- 控制论：双向调节——人类和 Agent 互为控制器和被控对象
- 有限理性：满意化——审批 = 人类判断"够好了"的信号
- 认知：协同重映射——alignment 不是控制，是共享空间的双向适应

**健康指标**：
- REQUIRE_APPROVAL → Agent 报告拦截 + 审批 ID + Dashboard 链接
- Dashboard 审批后 replayToolCall 自动执行
- 审批通过/拒绝率反馈到 constraintEffectiveness
- 人类审批决策影响 Agent 后续行为（学习）

**病征模式**：
| 测试症状 | 理论解释 | 维度内改进方向 |
|----------|----------|---------------|
| Agent 被拦截后不报告审批 ID | 有限理性: Agent 无法消费拦截信息 | 工具错误消息结构化 |
| 审批后 Agent 不知道已通过 | 控制论: 反馈回路断裂 | 审批结果主动通知 Agent |
| 人类始终通过（100%） | 认知: 协同退化为橡皮图章 | 约束效能建议"放宽或删除" |
| 人类始终拒绝（100%） | 认知: Agent 未学习人类偏好 | Agent memory 记录拒绝原因 |

**当前检查点**：B4 管理闭环（B4.15-B4.19）, S3.08（审批 API）

### Σ11 能力匹配度（MATCH）

**定义**：基础设施对模型能力的假设是否匹配现实——既不过度制约（friction > value），也不支持不足（model can do more but infra can't express）。

> **为什么需要独立维度**：Σ1-Σ10 评估"基础设施是否正确工作"——隐含假设是"越完善越好"。但模型能力持续增长，曾经必要的机制可能变成不必要的摩擦，而模型新涌现的能力可能缺少基础设施支撑。这是一个贯穿所有维度的**元评估**——评估的不是机制本身，而是机制与当前模型能力之间的**阻抗匹配**。

**理论锚点**：
- OS：系统调用接口的表达力应匹配用户态程序的能力——接口太窄（under-support）或权限检查太严（over-constraint）都降低系统效率
- 控制论：**Ashby 必要多样性的逆定理**——当被控系统（Agent）的能力超过控制器（Management）的假设时，控制器不是提供安全保障，而是制造人为瓶颈
- 有限理性：随着模型"理性边界"扩展（更大上下文、更强推理），为有限理性设计的辅助工具（摘要、brief 模式）可能从"必要"变为"多余的间接层"
- 认知理论：基础设施 = 嵌入空间的骨架——如果骨架维度低于模型实际可以操作的维度，基础设施反而压缩了模型的导航能力

**两个评估方向**：

#### 方向 A：过度制约（Over-constraint）

基础设施施加的约束是否超过模型实际需要？

| 信号 | 含义 | 对应维度 |
|------|------|---------|
| 管理审批通过率 >95%（样本 ≥20） | 约束对该操作类型实质上是橡皮图章 | Σ3, Σ10 |
| Agent 持续绕过管理层（文件 I/O 替代 bps 工具） | 模型判断管理成本 > 直接操作成本 | Σ3, Σ9 |
| Workspace 指令中 Agent 已知的常识占比过高 | 指令浪费上下文窗口且未提供增量信息 | Σ6 |
| 二层路由中模型正确率已达 100% 但仍需显式路由 | 路由机制从"必要引导"退化为"冗余仪式" | Σ9 |
| 约束效能建议始终是"放宽"且无人执行 | Σ5 反馈信号被忽略——控制器未跟随被控系统成长 | Σ5, Σ3 |

#### 方向 B：支持不足（Under-support）

模型已具备但基础设施无法表达的能力。

| 信号 | 含义 | 对应维度 |
|------|------|---------|
| 模型输出高质量计划但无法通过工具一步执行 | 工具粒度太细，缺少组合/批量 API | Σ1, Σ7 |
| 模型能自评风险但无 API 表达置信度 | 基础设施未提供 confidence/self-assessment 通道 | Σ3, Σ10 |
| 不同能力模型使用同一套约束 | 基础设施不感知模型能力差异 | Σ3, Σ4 |
| 模型能生成结构化数据但必须走 YAML 编译 | 基础设施输入格式限制了模型的表达自由度 | Σ9, Σ8 |
| 模型尝试引用跨实体关系但 API 只支持单实体查询 | 基础设施关系模型弱于模型的推理能力 | Σ2, Σ6 |

**健康指标**：
- 管理审批通过率在 30%-90% 区间（非 0% 也非 100%）
- Agent 管理绕过率 <10%（文件 I/O 替代 bps 工具的比例）
- Workspace 指令 token 效率：增量信息 / 总 token ≥ 0.5
- 工具调用/业务产出比的跨模型方差 <2x（不同模型效率差异不应太大）
- 基础设施 API 覆盖了模型 >80% 的常见操作模式

**病征模式**：
| 测试症状 | 理论解释 | 维度内改进方向 |
|----------|----------|---------------|
| 强模型反复绕过管理层 | Ashby 逆定理: Agent 能力超过控制器假设 | 按模型能力分级约束（trust tier）|
| 弱模型失败而强模型 PASS | OS: 接口对弱模型太复杂 | 工具层面的能力自适应（简化模式）|
| 审批通过率始终 >95% | 控制论: 控制器与目标失配 | 自动执行 Σ5 建议的约束放宽 |
| 模型产出质量高但工具调用量异常高 | 有限理性: 不必要的间接层 | 工具合并或批量 API |
| 模型尝试未注册的操作模式 | 认知: 嵌入空间维度不足 | 新增工具或放宽现有工具参数 |

**与 R4-R7 基准测试的关联**：
- R7 GPT-5.4 (8.55) vs GLM-5 (1.30) → 7.25 分差距中有多少是模型差异、多少是基础设施对弱模型的支持不足？
- R6-R7 管理绕过在所有模型中复现（文件 I/O 替代 bps_update_entity）→ 系统性 over-constraint 信号
- R7 Claude 164 次工具调用但 0 管理触发 → 可能的 under-support：模型知道该做什么但工具不支持直接表达

**当前检查点**：AEF E11.*（能力匹配度补充测试）

## 三、诊断方法论

### 3.1 输入与输出

```
输入：E2E 测试报告（PASS/FAIL/WARN 清单 + Agent session JSONL）
        ↓
步骤 1：检查点 → 维度映射（查映射总表）
步骤 2：维度健康度计算
步骤 3：四元诊断（对每个不健康维度应用四个理论透镜）
步骤 4：交叉验证（同一问题出现在 ≥2 个理论 → 高置信度）
步骤 5：产出
        ↓
输出 A：当前能力结构维度下需改进的问题（维度内优化）
输出 B：新增结构能力建议（新维度或现有维度扩展）
```

### 3.2 维度健康度计算

对每个维度 Σk，计算：

```
health(Σk) = (PASS_count) / (PASS_count + FAIL_count + WARN_count)

分级：
  ≥ 0.95  →  HEALTHY（绿色）
  ≥ 0.80  →  ADEQUATE（黄色）
  ≥ 0.60  →  DEGRADED（橙色）
  < 0.60  →  CRITICAL（红色）
```

FAIL 权重高于 WARN——一个 FAIL 的影响 ≈ 3 个 WARN：

```
weighted_health(Σk) = PASS / (PASS + 3×FAIL + WARN)
```

### 3.3 四元诊断模板

对每个 DEGRADED 或 CRITICAL 维度，依次应用四个诊断问题：

```markdown
## Σk [维度名] — weighted_health = X.XX (DEGRADED)

### 失败检查点
- [check_id]: [FAIL/WARN] — [观测描述]

### OS 诊断：机制是否正确实现？
- [ ] 状态转移是否完备？
- [ ] 资源隔离是否有效？
- [ ] 权限控制是否生效？
→ 结论：[是/否，具体问题]

### 控制论诊断：机制是否有效调节？
- [ ] 是否存在闭环反馈？
- [ ] 反馈延迟是否可接受？
- [ ] 控制器多样性是否匹配系统多样性？
→ 结论：[是/否，具体问题]

### 有限理性诊断：机制是否对 Agent 可用？
- [ ] 信息呈现是否支持快速决策？
- [ ] 是否存在搜索-执行鸿沟？
- [ ] 是否有满意化信号？
→ 结论：[是/否，具体问题]

### 认知理论诊断：是否同时支持导航与重映射？
- [ ] 已知空间内导航效率如何？（工具调用/产出比）
- [ ] 遇到新情境时能否重映射？（新 Skill/约束创建）
- [ ] 近临界状态是否维持？（不过刚不过柔）
- [ ] 层级嵌套是否正确？（高层约束，低层执行）
→ 结论：[是/否，具体问题]

### 交叉验证
出现在 ≥2 个理论的问题：
- [问题描述] → 置信度 HIGH，建议 [改进方向]
仅出现在 1 个理论的问题：
- [问题描述] → 置信度 MEDIUM，需进一步观测
```

### 3.4 新增结构能力检测规则

当以下模式出现时，表明需要**新的结构维度**或**现有维度的重大扩展**：

| 检测模式 | 理论来源 | 建议 |
|----------|----------|------|
| 同一根因导致 ≥3 个维度退化 | 控制论: 隐藏的公共耦合 | 新增一个横切维度 |
| Agent 行为与预期完全不同但测试全 PASS | 有限理性: 测试盲区 | 新增覆盖此行为模式的检查点 |
| 某维度 health 连续 3 轮未改善 | 认知: 重映射停滞 | 问题可能不在此维度——检查上游依赖 |
| Agent 创造了测试未覆盖的新能力 | 认知: Agent 自发重映射 | 将此能力纳入新维度或扩展现有维度 |
| 管理拦截率 >50% 但全部被通过 | 控制论: 控制器与目标失配 | 约束需要重新校准或分层 |
| 两个维度的检查点高度相关 | OS: 关注点未分离 | 考虑合并为一个维度 |
| 强模型评分显著高于弱模型但两者均通过基础设施测试 | Σ11: 基础设施对能力差异不敏感 | 增加 MATCH 维度检查点 |
| 管理审批通过率 >95% 持续 3 轮 | Σ11 + Σ5: 约束未跟随被控系统成长 | 执行约束放宽或分级 |

### Stage 4.5: 能力匹配自适应（Σ11 MATCH）— ⬜ 未开始

### Stage 5: 集体智能（四理论协同）— ⬜ 未开始

系统作为集体智能体运行：
- ⬜ 多 Agent 协同的共享嵌入空间

## 附录 B：维度间依赖关系

```
Σ3 GATE ──→ Σ4 STAB ──→ Σ5 FDBK
   │              │            │
   │              ▼            ▼
   │         Σ10 COADAPT ←─ Σ5 FDBK
   │
   ▼
Σ9 HIER ──→ Σ8 EVOL
   ▲
   │
Σ1 PROC ←─ Σ7 SCHED
   │
   ▼
Σ2 ENTITY ─→ Σ6 INFO

            Σ11 MATCH
           ╱    │    ╲
          ▼     ▼     ▼
      Σ3 GATE  Σ6 INFO  Σ9 HIER
          │           │
          ▼           ▼
      Σ8 EVOL   Σ10 COADAPT
```

读法：箭头表示"依赖"——Σ4 的健康依赖 Σ3 正确工作。如果上游维度 CRITICAL，下游维度的诊断应首先排除上游原因。

Σ11 MATCH 是元维度——不依赖其他维度，但其评估结果影响 Σ3/Σ6/Σ8/Σ9/Σ10 的诊断解读。当 Σ11 提示 over-constraint 时，Σ3 GATE 的"高覆盖率"不再是正面指标；当 Σ11 提示 under-support 时，Σ8 EVOL 的"高 Skill 创建数"可能是模型在弥补基础设施不足。
