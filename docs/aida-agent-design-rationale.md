## 一、为什么用户只应有一个日常交互对象

### 问题

AIDA 系统中有多个 Agent（BPS-Expert、Org-Architect，以及蓝图中动态创建的业务 Agent）。用户应该直接与谁对话？

### 推理

**认知带宽论证**：人类的认知带宽有限。与一个固定助理保持上下文同步已经很昂贵，维护多个主体的上下文会迅速耗尽注意力。这是人类认知的硬约束，不是软件设计可以优化掉的。

因此引入 Aida 作为唯一日常对话伙伴。Aida 代替人类承担了"与内部系统保持上下文同步"的认知负担。用户只需向一个实体表达意图，由 Aida 负责理解、分解、调度、翻译、反馈。

### 架构含义

Aida 本质上就是三层架构中"智能编排层"的人格化——不是抽象的调度引擎，而是一个具有上下文、判断力和对话能力的主体。

```
旧拓扑：[User] → [BPS-Expert] / [Org-Architect]  (平行独立，用户需同时维护多段对话)
新拓扑：[User] ↔ [Aida] → [BPS-Expert] / [Org-Architect]  (层级管理，用户只维护一段对话)
```

### 双通道互补

用户通过两个互补通道与系统交互：
- **Aida（自然语言）**：高频、模糊、探索性交互
- **Dashboard（结构化）**：全景状态展示 + 紧急操控/强制干预

两者不冲突——自然语言适合日常推动，Dashboard 适合需要精确操作的干预场景。

### Aida 超越"工具"的关键能力

1. **主动沟通**：不是被动响应，而是主动汇报异常、机会、需要用户决策的关键点。这是"管理助理"与"工具"的核心区别。
2. **上下文翻译**：用户说业务语言（"签约后自动安排调研"），系统跑技术语言（`process_state == 'TERMINATED'`）。Aida 负责双向翻译，用户永远不需要接触技术细节。
3. **升级判断**：哪些自主处理、哪些汇报但不等指令、哪些必须等用户决策——这个判断本身是 Aida 的核心能力，并应随协作深入逐步校准。

## 二、"BPS 是结晶机制，不是操作系统"

### 问题

BPS-Expert 和 Org-Architect 都很"重"——它们把动态涌现的计算任务"固化"了。所有流程都需要经过 BPS 吗？

### 核心论证

大模型（如 Opus 4.6）已具备 Agent team 能力，OpenClaw 有 Multi-Agent 机制。很多内部过程没必要呈现给人类，更没必要全部形式化为 BPS 蓝图。

**随着 LLM 能力增长，形式化的价值门槛在提高**。以前需要显式工作流定义的事情，现在可以通过模型的原生推理和工具调用能力处理。

因此得出核心原则：

> **BPS 是结晶机制，不是操作系统。**

大多数时候 Aida 用自身推理能力动态规划执行，只在特定条件下才将流程"结晶"为 BPS 蓝图。这将 BPS 从"所有事情必须经过的管道"重新定位为"按需激活的结构化工具"。

### 架构重构

```
旧理解：BPS 层（定义所有流程）→ 编排层（调度）→ Agent 层（执行）
新理解：Aida（动态规划 + 判断 + 对话）← 核心主体
         ├─ 直接调度 Agent/工具         ← 大多数情况
         ├─ BPS 结晶                    ← 满足特定条件时启用
         │   ├─ BPS-Expert（协助建模）
         │   └─ 蓝图 → 引擎 → 流程状态
         ├─ Org-Architect              ← 需要正式创建/部署 Agent 时
         └─ Dashboard                  ← 状态可视化 + 人类干预
```

**关键转变**：Aida 不是 BPS 之上的 UI 层，而是整个系统的主体。BPS 是她工具箱里的一件重型工具。

### 元判断能力

"何时应该使用 BPS"这个判断本身是一种**元判断能力**——不是"怎么做某事"，而是"该用什么方式做这件事"。这区分了一阶知识（领域专业技能）和二阶知识（知道何时形式化、何时保持灵活、何时让用户参与、何时自主行动）。

这组元判断能力被写入 Aida 的 SOUL.md 作为核心行为准则之一。

## 三、七个结晶化场景的推导

前四个场景由用户提出，后三个由 AI 补充。每个场景都有明确的推理依据。

### 场景 1：协作协议/护栏（用户提出）

**推理**：人类的意志/意图需要在 Aida 的自主行为中持久生效。例如"发布前必须经我审批"——不能靠 Aida 每次"记住"，需要固化为规则。

**为什么需要 BPS**：BPS Rule 正好承担这个角色。人类的常设指令变成蓝图中的 `manual` 节点，确保不被绕过。

### 场景 2：人类可见性需求（用户提出）

**推理**：人类无法参与他们看不到的流程。当流程的中间环节需要人类判断或操作时，该流程必须有可见的状态表示。

**为什么需要 BPS**：需要 Dashboard 可视化展示流程状态，这要求流程在引擎中有持久化的状态表示。

### 场景 3：超越上下文的持久性（用户提出）

**推理**：LLM 上下文窗口和会话生命周期是有限的。跨天、跨周的任务如果不持久化，会话一断状态就丢了。

**为什么需要 BPS**：BPS Process 的 SQLite 持久化解决这个问题——状态不依赖于任何会话的存续。

### 场景 4：Aida 自主判断空间（用户提出）

**推理**：弹性空间。Aida 可以根据上下文判断什么时候需要结构化。例如同一个流程她已经手动执行了三次以上，可以建议固化。

**为什么保留这个弹性**：避免规则体系过于刚性，允许 Aida 在实践中发现新的结晶化时机。

### 场景 5：可审计性要求（AI 补充）

**推理**：不是所有决策都需要追溯，但当决策涉及资金、合规或外部承诺时，决策链必须可追溯。

**为什么需要 BPS**：BPS 的 `eventsTriggeredLog` 和状态迁移记录天然提供审计链——每个规则何时被评估、为什么触发、执行了什么指令，全部有记录。

### 场景 6：可复用性（AI 补充）

**推理**：当一个流程经过反复打磨已经成熟，且预期会重复执行时，固化为蓝图是合理的。BPS 充当"最佳实践固化"——类比将临时脚本升级为正式程序。

**时序考量**：早期不应过早形式化。只有经过"反复使用打磨"的流程才应成为正式蓝图。过早结晶化等于过早优化，会限制灵活性。

### 场景 7：多方协调（AI 补充）

**推理**：当任务涉及多个人或外部系统需要共享状态进行协调时，Aida 的对话式记忆不够——需要外化的状态表示作为协调基础。

**为什么需要 BPS**：BPS 提供统一的流程状态共享机制，多方都可以通过引擎查询当前状态。

## 四、协作边界设计

### 为什么 BPS-Expert 和 Org-Architect 保持独立 Agent

三个理由：
1. **上下文窗口效率**：每个专业 Agent 有自己的上下文预算，不与 Aida 的对话上下文竞争
2. **专业化**：深度领域知识集中在专用 Agent 中，SOUL.md 可以装载大量 BPS 理论
3. **可测试性**：每个 Agent 可以独立测试

### 为什么允许 BPS-Expert ↔ Org-Architect 直接通信

当 BPS-Expert 识别出蓝图中需要 Agent 服务（`executorType: agent`）时，需要向 Org-Architect 提出 Agent 需求。如果每次都通过 Aida 中转，会增加不必要的延迟和上下文成本——这本质上是一个技术交接。

### 为什么 Aida 必须被通知结果

Aida 是与用户保持上下文同步的实体。如果她不知道 BPS-Expert ↔ Org-Architect 的交互，她就无法：
1. 准确向用户汇报状态
2. 对后续步骤做出知情决策
3. 兑现"唯一交互伙伴"的承诺

### 权限模型

- **Org-Architect**：`allowAgents: ["*"]`——因为它需要动态创建新 Agent
- **Aida**：`allowAgents: ["bps-expert", "org-architect"]`——显式列表，因为下属集合是稳定已知的
- **Aida** 在 OpenClaw 中应作为 HOST agent，而非某个系统的下属

## 五、其他设计决策

### Org-Architect 命名

原名 Meta-Architect，因"Meta"太抽象不能清晰传达职责而重命名。评估了四个候选：
- **Org-Architect**（组织架构师）：强调构建 Agent 组织拓扑为核心使命 ✓ 选中
- Agent-Ops（Agent 运维）：强调生命周期运维
- Agent-Forge（Agent 锻造厂）：强调创造性
- Crew-Architect（团队架构师）：强调协作团队管理

选择 Org-Architect 是因为它与 BPS Expert 形成清晰的职责对比：一个管业务流程，一个管组织结构。

### 平台名与 Agent 名的统一

Aida 同时作为平台名（AIDA = Agile Intent-Driven Architecture）和 Agent 名。Agent 本质上就是这个架构的人格化，这是一个干净的设计选择。

### Workspace 放置

Aida 的 workspace 放在与其他 Agent 相同的目录结构中（`agents/aida/`），因为所有 Agent 作为 AIDA 平台的一部分一起部署。
