## 一、背景

AIDA 作为 Agent 基础设施平台，其单元任务规模受到多个物理约束条件的限制。本文档分析这些约束条件，并给出扩展决策框架。

## 三、Layer 2：模型上下文窗口约束

### 3.1 各模型上下文窗口

| 模型 | 上下文窗口 | 实际可用（扣除 System） | 推荐用途 |
|------|-----------|------------------------|----------|
| Claude Opus 4.6 | 200K | ~150K tokens | 复杂推理、长任务链 |
| GPT-5.4 | 128K | ~100K tokens | 通用任务 |
| Gemini 3.1 Pro | 1M+ | ~800K tokens | 超长上下文场景 |
| Qwen3.5-Plus | 131K | ~100K tokens | 性价比优先 |
| Kimi K2.5 | 256K | ~200K tokens | 长文档处理 |

### 3.2 AIDA 上下文占用分析

```
System Prompt 组成（估算）：
├── Tooling（14 BPS tools 定义）         ~2K tokens
├── Safety                               ~0.5K tokens
├── Skills（7 个，XML 列表）             ~0.5K tokens
├── Workspace Files（Bootstrap 7 文件）  ~20-40K tokens
│   ├── AGENTS.md                        ~5K tokens
│   ├── SOUL.md                          ~1K tokens
│   ├── TOOLS.md                         ~2K tokens
│   ├── HEARTBEAT.md                     ~0.5K tokens
│   ├── USER.md                          ~0.2K tokens
│   ├── IDENTITY.md                      ~0.2K tokens
│   └── MEMORY.md（动态）                 ~10-30K tokens
├── Session History                      动态增长
└── Compaction 触发阈值                  ~80% 窗口
```

### 3.3 约束表现

| 现象 | 原因 | 阈值 |
|------|------|------|
| 任务链断裂 | 早期上下文被 Compaction | 链深度 > 5-7 层 |
| 业务域混淆 | 多域上下文交叉污染 | 并发业务域 > 3 |
| 关系网络丢失 | 实体关系超出窗口 | 关系节点 > 50 |

### 3.4 扩展策略

| 策略 | 触发条件 | 实现方式 | 效果 |
|------|----------|----------|------|
| **新建 Agent workspace** | 业务域独立 + 需隔离人格 | `agent-create` Skill | 完全隔离，独立上下文 |
| **Sub-Agent 拆分** | 单次任务复杂度高 | `sessions_spawn` | 任务级隔离，共享 workspace |
| **Skill 结晶** | 重复模式 > 3 次 | `skill-create` Skill | 压缩上下文，复用模式 |

## 五、Layer 1：数据级约束

### 5.1 SQLite 约束

| 约束 | 阈值 | 影响 | 监控 |
|------|------|------|------|
| 单表行数 | < 10M | 查询性能下降 | `SELECT COUNT(*)` |
| 数据库文件大小 | < 10GB | 备份/恢复时间 | `ls -lh bps.db` |
| 并发写入 | < 1000 TPS | 写入延迟 | WAL 模式监控 |
| 长事务 | < 1s | 阻塞其他写入 | 事务日志 |

**当前数据模型**：

```sql
bps.db
├── bps_processes（任务）        -- 增长最快
├── bps_dossiers（实体）         -- 持续增长
├── bps_dossier_versions（版本） -- 历史数据
├── bps_management_*（管理）     -- 相对稳定
├── bps_collaboration_tasks（协作）-- 新增
└── bps_skill_metrics（指标）    -- 日志型
```

### 5.2 向量索引约束

**内存占用估算**：

```
单文档向量：~400 tokens × 1536 维 × 4 bytes ≈ 2.4 MB
1000 个文档 ≈ 2.4 GB 内存
10000 个文档 ≈ 24 GB 内存
```

**约束表现**：
- 内存受限时，语义搜索性能下降
- 索引构建时间随文档量增长
- 冷启动时索引加载时间长

### 5.3 文件系统约束

**Workspace 文件增长**：

```
~/.openclaw/workspace/
├── MEMORY.md（长期记忆）         -- 缓慢增长
├── memory/
│   ├── 2026-03-01.md            -- 日增 ~5-10KB
│   ├── 2026-03-02.md
│   └── ...（持续累积）
└── skills/
    └── ...（7+ skills，相对稳定）
```

**约束表现**：
- 日志型文件持续增长
- inode 耗尽（极端情况）
- 备份/恢复时间增长

### 5.4 扩展策略

| 策略 | 触发条件 | 实现 |
|------|----------|------|
| **分表** | 单表 > 10M 行 | 按时间/项目分表 |
| **分库** | 数据量 > 10GB | 按项目分库 `~/.aida_{project}/` |
| **迁移 PostgreSQL** | 并发写入 > 1000 TPS | 改造 Store 层 |
| **向量索引分片** | 文档量 > 10K | 外部向量库（Milvus/Qdrant） |
| **日志归档** | memory/ > 1GB | 压缩归档 + 冷存储 |

## 七、压缩机制

### 7.1 Skills：上下文压缩

**原理**：将复杂操作模式压缩为单一指令。

```
原始操作序列（~500 tokens）：
  1. 读取项目配置
  2. 分析业务需求
  3. 设计数据模型
  4. 生成实体定义
  5. 创建 YAML 蓝图
  6. 调用 bps_load_blueprint
  7. 验证加载结果

Skill 压缩后（~50 tokens）：
  "使用 blueprint-modeling Skill 完成建模"
```

**压缩比**：~10:1

### 7.2 Agent Workspace：认知隔离

**原理**：隔离上下文，避免相互污染。

```
单 Agent 多业务域（上下文污染）：
  ┌─────────────────────────────────────┐
  │  Agent: aida                        │
  │  ├── 上下文：业务域A + B + C         │
  │  ├── 实体：100+（混合）              │
  │  └── 状态：混乱                      │
  └─────────────────────────────────────┘

多 Agent 隔离（认知清晰）：
  ┌──────────────────┐  ┌──────────────────┐
  │  Agent: aida     │  │  Agent: assistant │
  │  ├── 域：A       │  │  ├── 域：B        │
  │  ├── 实体：30    │  │  ├── 实体：40     │
  │  └── 状态：清晰  │  └── 状态：清晰     │
  └──────────────────┘  └──────────────────┘
```

### 7.3 Gateway 实例：执行隔离

**原理**：隔离资源竞争，保证 SLA。

```
单 Gateway（资源竞争）：
  ┌─────────────────────────────────────┐
  │  Gateway                            │
  │  ├── Agent A（高负载）               │
  │  ├── Agent B（饥饿）                │
  │  └── 事件循环：阻塞                  │
  └─────────────────────────────────────┘

多 Gateway（资源隔离）：
  ┌──────────────────┐  ┌──────────────────┐
  │  Gateway-A       │  │  Gateway-B        │
  │  ├── Agent A     │  │  ├── Agent B      │
  │  └── 独立事件循环 │  └── 独立事件循环    │
  └──────────────────┘  └──────────────────┘
```

## 九、总结

### 9.1 约束条件速查表

| 层级 | 约束 | 阈值 | 扩展单位 | 决策点 |
|------|------|------|----------|--------|
| **Layer 3** | OpenClaw Gateway 带宽 | Agent > 20，延迟 > 500ms | Gateway 实例 | 执行吞吐 |
| **Layer 2** | 模型上下文窗口 | 使用率 > 80%，业务域 > 3 | Agent Workspace | 认知复杂度 |
| **Layer 1** | SQLite 写入/存储 | 单表 > 10M，延迟 > 100ms | 数据库实例 | 存储容量 |

### 9.2 核心洞察

1. **Skills 是"上下文压缩"机制**：将复杂操作模式压缩为单一指令，压缩比 ~10:1
2. **Agent Workspace 是"认知隔离"机制**：隔离上下文，避免多业务域相互污染
3. **Gateway 实例是"执行隔离"机制**：隔离资源竞争，保证 SLA

### 9.3 当前建议

基于 AIDA 当前阶段（功能测试 → 业务测试）：

| 优先级 | 行动 | 原因 |
|--------|------|------|
| P0 | 实现 Agent 级监控 | 预警上下文瓶颈 |
| P1 | 制定 Skill 结晶规则 | 自动压缩重复模式 |
| P2 | 设计多 Gateway 架构 | 为生产部署做准备 |
