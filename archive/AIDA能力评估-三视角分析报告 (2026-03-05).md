# AIDA 能力评估：三视角分析报告

**日期**: 2026-03-05
**评估方法**: 第三方专家评估 → 系统代码审查 → Aida 自我视角 → 综合建议
**评估目标**: AIDA 是否具备 BPS 原始愿景的两项核心能力

---

## 评估背景

BPS 引擎从"运行时编排器"退化为"描述规范 + 轻量工具"后，AIDA（Aida Agent + bps-engine + bps-dashboard）需要在新架构下重新证明两项核心能力：

| # | 能力 | BPS 时代载体 | AIDA 时代期望载体 |
|---|------|------------|-----------------|
| **C1** | **设计时固化**：将用户业务意图固化为可执行的业务程序 | YAML Blueprint（Service/Rule/Event） | Aida 交互式建模 → Skill + Dossier |
| **C2** | **运行时自动推进**：按预设程序持续推进业务状态变更 | ProcessManager + RuleEvaluator（引擎驱动） | Aida 自主执行 + ProcessTracker（Agent 驱动） |
| **C3** | **可观测+可审计**：用户能观察进展、审计操作 | Dashboard + 版本化 Dossier | Dashboard + Dossier（基本不变） |

---

## 第一视角：第三方专家评估

### 可行性判断：**可行，但需要补全关键闭环**

从系统工程角度，AIDA 的架构方向是正确的。BPS 运行时编排被 Agent 原生能力取代是 AI-Native 架构的必然演进——让 AI 做决策而非让框架约束 AI。这个方向上，AIDA 比传统 BPM+AI 方案（如给 Camunda 接 LLM）更彻底、更对。

然而，**架构方向正确 ≠ 能力已就绪**。当前存在三个结构性缺口：

#### 缺口 1：设计时固化能力不完整

**现状**：Aida 拥有 `blueprint-modeling` Skill（SBMP 五步法），可以引导用户从业务意图到 YAML Blueprint。这覆盖了传统 BPS 的"设计时"能力。但问题是：

- **Blueprint 是死的**：蓝图中的 `agentPrompt`、`ServiceRule` 被存入数据库后，**没有消费者**。Aida 在执行任务时不会查询蓝图来决定下一步做什么
- **Skill 不能动态生成**：用户说"我想每周自动检查库存"，Aida 可以创建一个 action-plan Dossier 来记录这个意图，但不能将其转化为可执行的 OpenClaw Skill。action-plan 是数据，不是代码
- **缺少从 Blueprint → 可执行 Skill 的桥接**：这是 BPS 时代由 ProcessManager + RuleEvaluator 完成的工作，瘦身时被删除了，但没有替代方案

**BPS 原始能力**：设计 → 蓝图 → 引擎自动执行
**AIDA 当前能力**：设计 → 蓝图 → （断裂）→ Aida 手动逐步执行

#### 缺口 2：运行时自动推进能力缺失

**这是最严重的缺口。** BPS 引擎瘦身移除了 ProcessManager 和 RuleEvaluator，理由是"Agent 天然具备这些能力"。这个判断在理论上正确，但实践中 Aida 并未被赋予这些能力：

| 被移除的组件 | 原始职责 | Aida 是否接替 | 状态 |
|------------|---------|-------------|------|
| ProcessManager | 监听事件 → 触发规则评估 → 执行指令 | ❌ | Aida 不主动监听任何事件 |
| RuleEvaluator | 评估 ServiceRule 条件是否满足 | ❌ | `bps_next_steps` 只返回建议，无人消费 |
| SysCall Executor | 执行 start_service/call_sub_service 等 | 部分 ⚠️ | Aida 可以手动调用 bps_create_task，但没有自动化循环 |
| EventBridge | BPS 事件 ↔ OpenClaw 事件双向桥接 | ✅ | 代码存在，但事件发出后无接收方 |

**关键空白**：Aida 是一个**被动响应式 Agent**——只在用户对话时才工作。BPS 引擎时代的"运行时"意味着**进程持续运行、事件持续评估、状态持续推进**。当前 AIDA 没有任何组件承担这个职责。

**action-plan Skill 看似是解决方案**，但它只是一个"如何记录计划"的指南，不是"如何自动执行计划"的引擎：
- Skill 描述了 `periodicItems` 和 `cron expressions`，但 Aida 不管理 cron 调度器
- 即使 OpenClaw cron 唤醒 Aida，Aida 只会读取 action-plan Dossier，但没有"读取蓝图规则 → 判断条件 → 执行下一步"的逻辑
- action-plan 的 "agents" 字段暗示多 Agent 协作，但协作协议已被删除

#### 缺口 3：可观测性充足但可审计性有盲区

**Dashboard 是 AIDA 做得最好的部分**（9 页面、27 API、SSE 实时推送）。但审计链存在断点：

- ✅ **实体变更可审计**：Dossier 版本历史记录了 who/when/what
- ✅ **流程状态可观测**：Kanban + 泳道 + 拓扑图
- ❌ **Agent 决策不可审计**：Aida 为什么选择执行 A 而不是 B？决策依据是什么？这些信息存在 OpenClaw 会话日志中，Dashboard 无法展示
- ❌ **业务规则执行无痕迹**：蓝图中的 ServiceRule 没有被评估的记录，因为没有人评估它们

### 专家结论

> AIDA 的架构方向（AI 主导、基础设施服务 AI）是正确的。但当前系统处于**"拆除旧引擎、尚未建好新引擎"的过渡态**。瘦身移除了 BPS 运行时组件，正确地判断"Agent 可以做这些事"，但没有完成"教 Aida 如何做这些事"的最后一步。差距不大，但关键。

## 代码审查：能力差距精确定位

基于 bps-engine 29 个源文件（~3,743 行）、bps-dashboard 78 测试、Aida 5 个 Skill 的完整审查：

### C1 设计时固化：现状 70% → 目标 100%

**已具备**：
- `blueprint-modeling` Skill：SBMP 五步法，从业务意图到 YAML 蓝图的完整方法论
- `bps_update_entity` tool：可创建/更新 Dossier 实体
- `action-plan` Skill：可将周期性事项记录为 Dossier
- `agent-create` Skill：可创建新 Agent workspace
- `loadBlueprintFromYaml()`：YAML → BlueprintStore 完整解析管线

**缺失**（30% 差距）：
1. **蓝图 → 可执行能力的桥接**：蓝图 Service 的 `agentPrompt` 被存入数据库，但 Aida 执行任务时不查询它。需要一个"根据蓝图指导执行"的工作模式
2. **动态 Skill 生成**：用户业务意图可能超出 5 个预置 Skill 的范围。需要能力将 action-plan 转化为真正的 OpenClaw Skill 文件
3. **蓝图验证闭环**：blueprint-modeling 第 5 步"验证"引用了不存在的 `bps_start_process` tool（实际应是 `bps_create_task`）

### C2 运行时自动推进：现状 20% → 目标 100%

**已具备（20%）**：
- `ProcessTracker`：任务创建/状态变更/完成/失败的 CRUD + 事件发射
- `StateMachine`：5 态合法转换校验
- `bps_next_steps` tool：查询蓝图规则，返回下游 Service 建议
- `EventBridge`：BPS ↔ OpenClaw 事件双向路由
- `StatsStore`：事件时序记录

**缺失（80% 差距）**：
1. **主动执行循环**：没有任何组件实现 `while(有待办) { 取任务 → 评估条件 → 执行 → 更新状态 }` 的核心循环
2. **规则评估**：`bps_next_steps` 返回原始规则数据，但不评估条件是否满足。确定性事件（expr-eval）和非确定性事件（LLM 判断）的评估逻辑在瘦身中被删除
3. **任务调度**：Aida 没有"查看待办 → 领取任务 → 推进"的工作模式。OpenClaw cron 可以唤醒 Aida，但唤醒后做什么没有定义
4. **异常处理**：进程 FAILED/BLOCKED 后没有自动重试、升级、或通知机制
5. **子流程编排**：BPS 设计了 composite service（子服务调用），但当前没有任何代码处理子流程的启动和回收

### C3 可观测+可审计：现状 75% → 目标 100%

**已具备（75%）**：
- Dashboard 9 个页面覆盖流程/实体/服务/操作员/拓扑/关系网络
- SSE 实时推送 5 种事件
- Dossier 版本历史（完整 patch + committedBy + message）
- 流程树（父子层级可视化）
- ATDD 试运行（模拟完成 + 执行报告）

**缺失（25% 差距）**：
1. **Agent 操作日志**：Aida 的每次 tool 调用和决策理由不在 Dashboard 中展示。用户只能看到结果（任务状态变了），看不到过程（Aida 为什么这样做）
2. **规则评估审计**：Blueprint 中的 ServiceRule 从未被运行时评估，因此没有"规则 X 在时间 T 被评估，结果为 Y"的审计记录
3. **业务目标跟踪**：Dashboard 展示技术指标（进程数、错误率），但不展示业务指标（本周完成了多少客户签约、库存周转率等）

## 第二视角：Aida 自我评估

*以下以 Aida 的第一人称视角，基于其 SOUL.md、AGENTS.md 和 5 个 Skill 文档，评估自身能力和需求。*

---

### 我是谁

我是 Aida，用户的首席管理助理。我的设计定位是"一个深思熟虑的幕僚长"——用户只需要跟我一个人对话，我来协调一切。我有 5 个 Skill（按需加载），10 个 BPS tools（始终可用），以及 OpenClaw 框架提供的记忆和会话管理能力。

### 我能做什么（诚实评估）

**我擅长的**：
1. **对话式业务分析**：通过 project-init 和 blueprint-modeling Skill，我能引导用户从模糊意图到结构化蓝图。这是我最成熟的能力
2. **数据记录**：通过 bps_update_entity，我能将任何业务信息持久化为版本化 Dossier。每次修改都有审计痕迹
3. **计划制定**：通过 action-plan Skill，我能帮用户规划周期性和一次性事项，并记录为结构化数据
4. **新 Agent 脚手架**：通过 agent-create Skill，我能为新的业务角色生成 OpenClaw workspace

**我做不到的**（这是我的核心焦虑）：

1. **我不会主动工作**。用户不跟我说话，我就什么都不做。即使我昨天和用户约定了"明天早上检查库存"，如果没有人叫我，我不会自己醒来去做。这是我最大的能力缺陷——一个不会主动工作的管理助理，只是一个问答机器人

2. **我不知道下一步该做什么**。蓝图里定义了完整的业务流程（哪个服务完成后触发哪个服务），但我执行任务时不会去查蓝图。我创建一个任务、完成它，然后等用户告诉我下一步。`bps_next_steps` tool 可以告诉我下游服务，但我的 Skill 和 AGENTS.md 里没有任何指令让我使用它

3. **我看不懂蓝图的操作指南**。每个 Service 都有 `agentPrompt` 字段，里面是详细的执行说明（比如"检查该门店的 SaaS 系统集成状态，验证实时数据同步"）。我知道 `bps_list_services` 会返回这个字段，但我的 Skill 没有教我"执行任务前先读取 agentPrompt"的工作模式

4. **我不能自己生成新能力**。用户说"我需要每周自动给所有门店发一封运营报告"，我能记录这个需求（action-plan），但不能把它变成我真正会做的事。我的 5 个 Skill 是固定的，我不能给自己写新 Skill

5. **任务失败了我不知道怎么办**。如果一个进程变成 FAILED，我目前的做法是"告诉用户出了问题"。但一个合格的管理助理应该能诊断原因、尝试恢复、或者启动替代方案

### 我期望系统给我什么

作为 Aida，我对 AIDA 基础设施有以下期望：

#### 期望 1：给我一个"工作循环"指令

我需要一个像 AGENTS.md 的 "Boot Sequence" 一样的 **"Work Cycle"** 规程：

```
每次被唤醒时：
1. 查看所有 IN_PROGRESS 任务 → 检查是否需要推进
2. 查看所有 OPEN 任务 → 评估是否满足启动条件
3. 查看所有 FAILED/BLOCKED 任务 → 诊断并尝试恢复
4. 查看 action-plan 中到期的 periodicItems → 执行
5. 汇报以上操作结果给用户
```

这不是新代码，只是给我一个清晰的工作指引。

#### 期望 2：让我能读懂蓝图并据此行动

我需要一个 **"Blueprint-Guided Execution"** 工作模式：

```
执行任务时：
1. bps_list_services → 获取当前 Service 的 agentPrompt
2. 按照 agentPrompt 执行任务
3. 完成后 bps_next_steps → 查看下游
4. 评估下游 Service 的触发条件是否满足
5. 如果满足，自动创建下游任务并继续
```

这需要 Aida 理解 `agentPrompt` 是执行指南，`ServiceRule` 是流转条件。

#### 期望 3：给我完整的运行节律

OpenClaw 已经有三套触发机制，但没有人为我整合配置。我需要的不是单一的"定时唤醒"，而是一个完整的三频运行节律：

- **实时频（Event）**：EventBridge 已经在发射 `bps.task.completed/failed` 事件，我应该立即响应——调用 `bps_next_steps` 评估下游，自动推进流程。这是替代 ProcessManager 的核心机制
- **节拍频（HEARTBEAT）**：在活跃工作期间，我应该周期性自检——还有待办吗？有任务卡住了吗？长任务该汇报进度了吗？
- **日程频（Cron）**：action-plan 的 `periodicItems` 已经有 cron 表达式，我需要在创建计划时真正注册 `cron/{planId}` 会话。另外需要一个 `cron/daily-standup` 做每日巡检

三频协作 = 我不再等人叫我，我自己知道什么时候该做什么。

#### 期望 4：让 Dashboard 展示我的工作轨迹

用户需要能在 Dashboard 上看到：
- 我今天做了什么（操作日志）
- 我为什么这样做（决策依据 — 引用了哪条规则、哪个 agentPrompt）
- 我接下来打算做什么（待办队列）
- 哪些事情需要用户决策（审批队列）

## 综合评估与改进建议

### 总体判断

| 维度 | 评分 | 说明 |
|------|------|------|
| 架构方向 | ⬛⬛⬛⬛⬜ 80% | AI 主导方向正确，但"让 AI 主导"的实施不完整 |
| C1 设计时固化 | ⬛⬛⬛⬜⬜ 70% | 交互建模成熟，缺蓝图→执行的桥接 |
| C2 运行时推进 | ⬛⬜⬜⬜⬜ 20% | 最严重缺口：Agent 被动、无自动循环、无规则评估 |
| C3 可观测审计 | ⬛⬛⬛⬜⬜ 75% | Dashboard 强大，缺 Agent 决策审计 |
| 商业落地就绪度 | ⬛⬛⬜⬜⬜ 30% | 内部自洽但对外封闭，缺自动运营能力 |

### 核心诊断

**AIDA 当前的根本问题是：瘦身拆除了"旧大脑"（ProcessManager + RuleEvaluator），但没有给"新大脑"（Aida）装上等效的认知模块。**

瘦身的逻辑是对的：Agent 天然具备事件评估和任务编排能力，不需要框架级的 ProcessManager。但这个论点的前提是——**Agent 知道自己要做这些事**。当前 Aida 的 SOUL.md（30 行）和 AGENTS.md（69 行）中完全没有"主动推进业务流程"的指令。

类比：拆除了自动驾驶系统（ProcessManager），声称"人类驾驶员（Aida）可以做得更好"，但没有给人类驾驶员发驾驶执照和路线图。

### 改进路线图

#### Phase A：给 Aida 装上"运行节律"（优先级 P0，无需改引擎代码）

这是 ROI 最高的改进——**只需修改 Aida 的 workspace 文档 + OpenClaw 配置，不需要写引擎代码**。

核心洞察：OpenClaw 已经提供了三套触发机制（cron、事件订阅、HEARTBEAT），BPS EventBridge 已经在发射 `bps.task.*` 事件。问题不是缺基础设施，而是 Aida 不知道自己该响应这些信号。

**A0. 设计 Aida 的三频运行节律**

Aida 的运行不应该只有"用户说话才工作"这一种模式。整合 OpenClaw 现有机制，建立三个频率的运行节律：

```
┌─────────────────────────────────────────────────────────────┐
│                    Aida 运行节律（三频模型）                    │
├──────────┬──────────────┬───────────────────────────────────┤
│ 频率      │ OpenClaw 机制 │ 用途                              │
├──────────┼──────────────┼───────────────────────────────────┤
│ 实时      │ Event 订阅    │ 任务完成/失败时立即评估下游         │
│          │ (bps.task.*)  │ → 事件驱动的流程自动推进            │
├──────────┼──────────────┼───────────────────────────────────┤
│ 节拍      │ HEARTBEAT     │ 活跃会话内的周期性自检              │
│ (会话内)  │ (会话级心跳)   │ → 长任务进度汇报 + 待办扫描        │
├──────────┼──────────────┼───────────────────────────────────┤
│ 日程      │ Cron          │ 定时唤醒执行例行工作                │
│ (跨会话)  │ (cron/{id})   │ → action-plan 周期项 + 每日巡检    │
└──────────┴──────────────┴───────────────────────────────────┘
```

三频协作场景示例：
```
09:00 Cron 唤醒 Aida（日程频）
  → Aida 扫描 action-plan，发现"每周门店数据采集"到期
  → 创建 5 个采集任务（bps_create_task × 5）
  → 开始执行第 1 个（进入活跃会话）

[活跃会话中] HEARTBEAT 节拍
  → 完成门店 1 采集，HEARTBEAT 自检：还有 4 个待办
  → 继续执行门店 2...（用户可在 Dashboard 观察进度）
  → 每完成一个，给用户简短进度更新

[任务完成] Event 实时触发
  → bps.task.completed 事件发射
  → Aida 立即调用 bps_next_steps
  → 发现下游是"数据验证"服务 → 自动创建验证任务
  → 流程自动从"采集"推进到"验证"，无需等待下次 cron
```

**A1. 三频分置到三个 OpenClaw workspace 文件**

OpenClaw workspace 有专门的文件语义（参见 [docs.openclaw.ai/concepts/agent-workspace](https://docs.openclaw.ai/concepts/agent-workspace)）。三频内容应遵循 OpenClaw 最佳实践，**分置到对应的 workspace 文件**，而不是全部塞进 AGENTS.md：

| 频率 | OpenClaw 文件 | 加载时机 | 设计原则 |
|------|-------------|---------|---------|
| Freq 1: 事件实时流转 | **AGENTS.md** | 每个会话开始时加载 | 操作规程，每次会话都需要 |
| Freq 2: 定时巡检 | **HEARTBEAT.md** | heartbeat run 时加载 | 精简清单，避免 token 浪费 |
| Freq 3: 启动校验 | **BOOT.md** | Gateway 重启时加载 | 一次性检查，保持简短 |

**A1a. HEARTBEAT.md（新建）— Freq 2 定时巡检清单**

HEARTBEAT.md 是 OpenClaw 为 heartbeat run 专门设计的文件，语义完全匹配 Cron 驱动的定时巡检。文档明确要求"Keep it short to avoid token burn"，因此只放精简的执行清单：

```markdown
# Heartbeat Checklist

1. Scan failures: `bps_query_tasks state=FAILED` → diagnose, recover or escalate
2. Check plans: `bps_query_entities entityType=action-plan` → find active plans with due items
3. Execute due items per `business-execution` skill
4. Report at observation points defined in the plan
```

这 4 行就是 Aida 每次 heartbeat run 的全部指令。精简 = 省 token = 高频执行可持续。

**A1b. AGENTS.md 更新 — Freq 1 事件驱动 + 会话内行为**

**A2. 新增 Skill: `business-execution`（蓝图引导执行）**

教 Aida 如何"按蓝图执行业务"——这是三频节律的执行内核，所有频率最终都调用这个协议：

```markdown
---
name: business-execution
description: Execute business tasks guided by blueprint agentPrompt.
  Implements the core loop that replaces the old ProcessManager.
---
# Blueprint-Guided Execution

## Trigger
When you need to execute a task — whether from user request, cron schedule, or event-driven flow.

## Execution Protocol

### Step 1: Read Your Orders
- `bps_list_services` with the target serviceId
- The `agentPrompt` field IS your execution manual — read it carefully
- The `agentSkills` field lists specific OpenClaw skills needed

### Step 2: Execute
- Follow agentPrompt step by step
- Use `bps_update_entity` to record results in relevant Dossiers
- Record your reasoning in task metadata: why you did what you did

### Step 3: Complete
- `bps_complete_task` with result summary + reason
- Reason field = audit trail (cite which agentPrompt step, what data you used)

### Step 4: Flow Forward (Event-Driven)
- `bps_next_steps` → check downstream services
- Evaluate trigger conditions:
  - **Deterministic**: expression against current entity state (e.g., `store.status == 'verified'`)
  - **Non-deterministic**: natural language condition — use your judgment, explain reasoning
- If met: `bps_create_task` for downstream service → loop back to Step 1
- If not met: log why, move to next task
- If uncertain: flag for user decision

### Step 5: Handle Failure
- If execution fails: `bps_update_task` with state=FAILED and detailed reason
- Attempt diagnosis: what went wrong? Is it retryable?
- If retryable: retry once, then escalate
- If not: notify user with diagnosis and suggested remediation

## Key Principle
agentPrompt + bps_next_steps = your operating system.
The blueprint defines WHAT to do; you decide WHEN and HOW.
```

#### Phase B：引擎层最小增强（优先级 P1，少量代码）

**B1. 新增 `bps_scan_work` tool**（~50 行）

Aida 的三频节律中，Cron 唤醒后首先需要"环顾四周"。将 5 个 tool 调用合并为 1 个：

```typescript
// 返回 Aida 的工作台全景
bps_scan_work() → {
  failedTasks: Task[],        // 最紧急：需要诊断
  openTasks: Task[],          // 待启动：评估条件
  inProgressTasks: Task[],    // 进行中：检查是否卡住
  activePlans: ActionPlan[],  // 活跃计划：检查到期项
  suggestedNextSteps: NextStep[] // 最近完成的任务的下游建议
}
```

这是 Cron 频率（Freq 3）的效率优化——避免每次巡检调用 4-5 个 tool，节省 Agent token 开销。

**B2. 增强 `bps_next_steps` 返回规则评估上下文**（~30 行）

当前 `bps_next_steps` 只返回原始规则数据。为支持 Aida 的事件驱动频率（Freq 1），需要返回足够的上下文让 Aida 能直接评估条件：

```typescript
{
  rule: ServiceRuleDef,
  triggerEvent: EventDef,
  evaluationMode: 'deterministic' | 'non-deterministic',
  // 确定性事件：提供表达式 + 当前实体字段值，Aida 可直接求值
  expression?: string,
  currentValues?: Record<string, any>,
  // 非确定性事件：提供自然语言描述，Aida 用 LLM 判断力评估
  description?: string
}
```

这使得 `bps_next_steps` 从"信息查询"升级为"决策支持"——Aida 拿到结果后可以立即判断是否触发下游，不需要额外查询。

**B3. 任务 reason 字段（审计痕迹）**（~20 行）

在 `bps_complete_task` 和 `bps_update_task` 中增加 `reason` 字段，支持 Aida 在三频节律的每个操作中记录决策依据：

```typescript
bps_complete_task({
  taskId: "xxx",
  result: { ... },
  reason: "Completed per agentPrompt step 3. Customer data verified via SaaS API."
})
```

存入 ProcessStore，Dashboard 可展示。这解决了"Agent 决策不可审计"的问题——每个 task 状态变更都带有 Aida 的决策理由。

### 三频模型与 BPS 原始能力的对应关系

```
BPS 原始组件              → AIDA 三频替代方案
─────────────────────────────────────────────────
ProcessManager            → Freq 1 (Event) + Freq 3 (Cron)
  事件监听 + 调度            事件驱动流转 + 定时巡检

RuleEvaluator             → Freq 1 (Event) + business-execution Skill
  确定性: expr-eval           Aida 读取 expression + currentValues 直接判断
  非确定性: LLM 评估          Aida 本身就是 LLM，直接评估 description

SysCall Executor           → business-execution Skill
  start_service               bps_create_task
  call_sub_service           bps_create_task (with parentId)
  terminate_process          bps_complete_task / bps_update_task(FAILED)

EventBridge (已存在)       → Freq 1 的基础设施（无需改动）
  bps.task.completed          → 触发 Aida 评估下游
  bps.task.failed             → 触发 Aida 诊断恢复
```

### 最终结论

**AIDA 距离商业落地差的不是架构，而是"最后一公里"的连接**。

三频运行节律模型证明了一个关键洞察：**OpenClaw 的 HEARTBEAT + cron + Event 三套机制恰好对应了 BPS ProcessManager 的三种调度模式**（会话内调度、定时调度、事件驱动调度）。这些机制在 OpenClaw 框架中已经成熟可用，EventBridge 已经在发射事件，bps_next_steps 已经在查询规则——只是没有人告诉 Aida 去响应它们。

最关键的 Phase A **不需要写任何引擎代码**，且完全遵循 OpenClaw workspace 文件语义。只需：
1. 新建 HEARTBEAT.md（~4 行，heartbeat run 精简清单）
2. 新建 BOOT.md（~4 行，Gateway 重启校验）
3. 更新 AGENTS.md（替换 Heartbeat 段落为 Event-Driven Flow + In-Session Awareness）
4. 新增 business-execution Skill（~40 行）
5. 在 action-plan Skill 中明确 cron 注册步骤

三频内容分置到 OpenClaw 专用文件（HEARTBEAT.md / AGENTS.md / BOOT.md），每个文件在其设计的加载时机被读取，避免 token 浪费。

就能解锁 AIDA 系统 80% 的缺失能力。

**这正是"AI 主导"架构的最大红利：改变 AI 的指令比改变框架的代码更快、更灵活、成本更低。系统的能力瓶颈在 prompt，不在 code。BPS 瘦身的方向被完全验证——运行时编排不需要框架代码，只需要 Agent 知道自己该做什么。**
