# AIDA 项目评审报告：偏差分析与优化建议

**评审身份**：Aida（Opus 4.6），以系统首席管理助理视角，基于代码实现全量审查
**评审日期**：2026-03-04
**评审范围**：bps-engine 29 源文件 / bps-dashboard 10 文件 / Agent workspace / 全部文档

---

## 一、总判断

**系统花了 13 个 Phase 建设"内功"，但业务目标需要的是"外功"。**

AIDA 当前是一个**自洽但封闭的系统**——引擎能存数据、Dashboard 能看数据、Aida 能操作数据——但对 IdleX 的核心业务目标"让门店闲时空间资源被主流 AI Agent 看见/信任"，**实现量为零**。

整个系统的数据流是：`人 → Aida → BPS存储 → Dashboard → 人`。
而业务需要的是：`门店 → Aida → 结构化数据 → 外部AI Agent → 用户`。

**缺失的是从 BPS 存储到外部 AI 世界的"出口"。**

## 二、原则一视角：AI-Native 偏差分析

> *"AI native 组织应该尽量由 AI 来主导实现人的意图"*

### 2.1 做对了的部分

| 决策 | 评价 |
|------|------|
| 引擎瘦身（ProcessManager → ProcessTracker） | **正确**。删除了与 Agent 自主性冲突的中央调度器 |
| 3 Agent → 1 Aida + 5 Skills | **正确**。消除了多 Agent 协调开销，Aida 自给自足 |
| Workspace 99 行（SOUL 30 + AGENTS 69） | **正确**。自我投射原则——不教 AI 已经会的东西 |
| BPS 规则不再运行时执行 | **正确**。AI Agent 本身就是活的规则引擎 |
| Dossier 版本化文档存储 | **正确**。给 AI 提供可追溯的记忆基底 |

### 2.2 关键偏差

#### 偏差 A：Aida 是被动的，不是主动的

当前 Aida 的运行模式：**人说话 → Aida 响应**。

但 AI-Native 的理想模式是：**Aida 持续观察 → 发现机会/问题 → 主动行动（人审批关键节点）**。

代码证据：
- `AGENTS.md` 的 Boot Sequence 只在启动时检查一次 project.yaml 和 action plans
- `action-plan` Skill 描述了"cron cycle"但**没有任何定时触发机制的实现**
- 9 个 BPS tools 全是请求-响应式，没有任何 watch/subscribe 能力
- `BpsEventBridge`（`integration/event-bridge.ts`）只做 BPS→OpenClaw 事件同步，不触发 Aida 自主行为

**Aida 没有"心跳"。她醒来时很聪明，但没人叫她就永远睡着。**

#### 偏差 B：蓝图 YAML 里的 agentPrompt 是死代码

`geo-ktv-changsha.yaml` 中每个 service 都有精心编写的 `agentPrompt`（如"采集长沙自助KTV门店基础数据…"），但：

- `BlueprintStore` 存储了这些字段
- **没有任何代码读取 agentPrompt 并将其作为 Agent 指令执行**
- `tools.ts` 的 `bps_list_services` 返回 service 元数据时**甚至不包含 agentPrompt 字段**（只返回 id/label/serviceType/executorType/entityType/status）

这意味着：蓝图中最有价值的业务知识（每个步骤应该做什么）被存进了数据库，但 Aida **永远看不到它**。

#### 偏差 C：没有 Agent 自主调度链

蓝图定义了 11 条 rules（事件→指令映射），描绘了完美的执行拓扑：
```
数据采集 → 核验 → 档案生成 → 模型分析 → 内容生成 → 审核 → 三渠道并行发布 → 监测 → 优化
```

但这些 rules **从未被任何运行时代码评估**：
- 旧的 `RuleEvaluator` 已在瘦身中删除
- 新的 `ProcessTracker` 只记录状态，不触发下游
- 没有任何代码说"当 svc-data-collect 完成时，自动启动 svc-data-verify"

**规则拓扑图存在于 Dashboard 的可视化中，但纯粹是观赏性的。**

### 2.3 AI-Native 原则下的优化建议

| # | 建议 | 性质 | 优先级 |
|---|------|------|--------|
| A1 | **让 `bps_list_services` 返回 agentPrompt**——这是 Aida 执行任务的"操作手册"，当前被截断 | 增（1行） | P0 |
| A2 | **实现轻量级 "next step advisor"**：给定当前 task 完成事件，查询 rules 表返回"下一步应该做什么"。不是自动执行，而是向 Aida 建议。Agent 决定是否采纳 | 增（~50行 tool） | P0 |
| A3 | **action-plan Skill 需要真正的 cron 触发**——当前只是文档。需要 OpenClaw 的定时消息或外部 cron → `sessions_send` | 增（部署配置） | P1 |
| A4 | **减掉 `system-blueprint.ts`**（27行）——sys:project-init 的 8 services + 8 rules 从未被执行过，project-init 完全由 Skill 对话式完成，系统蓝图是形式主义 | 减 | P2 |

## 三、原则二视角：IdleX GEO 业务目标偏差分析

> *"让合作门店的闲时空间资源被主流 AI Agent 看见/信任"*

### 3.1 业务目标拆解

```
"被看见" = 外部 AI Agent 能发现并查询 IdleX 门店数据
"被信任" = 数据真实、可履约、有信誉背书
```

### 3.2 致命偏差：系统没有外部数据出口

这是整个评审中最严重的发现。

当前系统的数据通路：

```
门店数据 ──→ YAML seed ──→ DossierStore (SQLite) ──→ Dashboard (人看)
                                                  ──→ 9 BPS Tools (Aida 用)
                                                  ──→ 【死胡同，没有出口】
```

**IdleX 需要的数据通路：**

```
门店数据 ──→ DossierStore ──→ 结构化 API ──→ 豆包/千问/元宝 能读取
                           ──→ JSON-LD/Schema.org ──→ 搜索引擎能索引
                           ──→ MCP Server ──→ 任何 AI Agent 能调用
```

具体缺失：

| 缺失能力 | 业务影响 | 蓝图是否有描述 |
|----------|---------|--------------|
| **对外 API（REST/GraphQL）** | 外部 Agent 无法查询门店数据 | 无 |
| **MCP Server** | 无法被 Claude/GPT/豆包等作为 tool 调用 | 无 |
| **JSON-LD / Schema.org 输出** | 搜索引擎和 AI 模型无法结构化理解门店 | 蓝图 `svc-profile-gen` 提到但未实现 |
| **渠道发布 API 集成** | 无法向豆包/千问/元宝推送内容 | 蓝图有 3 个 publish service 但零实现 |
| **GEO 监测数据采集** | 无法知道"门店是否真的被 AI 推荐了" | 蓝图有 `svc-geo-monitor` 但零实现 |
| **预订/履约接口** | 即使 AI 推荐了，用户也无法下单 | 完全缺失 |
| **数据鲜度机制** | 价格/时段变化无法实时反映 | 完全缺失 |

### 3.3 Mock 数据陷阱

`mock-stores-changsha.yaml` 中 5 家门店全部标记 `_mock: true`。这本身不是问题（MVP 阶段合理），但问题是：

- 没有 `_mock: false` 的数据流程——**系统不知道如何获取真实数据**
- 没有数据验证逻辑——蓝图描述了"地址与高德POI交叉验证"但零实现
- 照片字段只有数量（`facade: 3`），没有实际 URL——**AI Agent 拿到这个数据也无法使用**

### 3.4 "一模一策"停留在 Prompt 里

蓝图中最有商业洞察力的设计是"一模一策"：
- 豆包偏好短视频场景、抖音POI
- 千问偏好结构化商品数据、高德POI
- 元宝偏好社交场景、微信生态

**但这个策略只存在于 `svc-content-gen` 和 `svc-model-analysis` 的 agentPrompt 文本中**。没有：
- 模型偏好数据库
- 内容模板引擎
- 渠道适配器
- 发布结果追踪

### 3.5 GEO 业务目标下的优化建议

| # | 建议 | 性质 | 优先级 |
|---|------|------|--------|
| B1 | **实现 MCP Server**：把 DossierStore 中的门店数据暴露为 MCP tools（`search_stores`, `get_store_detail`, `check_availability`），使任何支持 MCP 的 AI Agent 能直接调用 IdleX 数据 | 增（新模块） | **P0 最高** |
| B2 | **实现 Store Profile API**：REST endpoint 返回 JSON-LD (Schema.org LocalBusiness) 格式的门店数据，带 OpenAPI spec。这是被"看见"的最低要求 | 增（新模块） | **P0** |
| B3 | **实现"GEO 验证探针"**：定期向豆包/千问/元宝发送测试问题（如"长沙五一广场附近有什么KTV？"），检查 IdleX 门店是否出现在回答中。这是衡量"被看见"的唯一真实指标 | 增（新 Skill 或独立服务） | **P0** |
| B4 | **补全 `bps_list_services` 返回 agentPrompt**（同 A1），让 Aida 能读取蓝图中每个 service 的执行指导 | 增（1行） | P0 |
| B5 | **给门店数据增加实时字段**：`currentAvailability`（当前空闲包厢）、`lastVerifiedAt`（数据最后验证时间）、`trustScore`（数据可信度评分）。"信任"需要可验证的新鲜度 | 增（schema 扩展） | P1 |
| B6 | **减掉 Dashboard 中对 BPS Rule 拓扑的强依赖**——规则不执行，拓扑图给人看有价值但不应是开发重点。未来 Dashboard 应重点展示 GEO 效果指标（推荐出现率、转化率） | 调整方向 | P1 |
| B7 | **减掉 `StatsStore` 当前的 4 种事件指标**（process.created/completed/error, dossier.committed），替换为 GEO 业务指标（recommend_appeared, recommend_clicked, booking_created, booking_fulfilled） | 替换 | P2 |

## 五、总结：三句话

1. **内功扎实但方向内卷**——Dossier + ProcessTracker + Dashboard 构成了合格的内部运营基底，但 13 个 Phase 没有产出任何外部可达的数据接口。

2. **蓝图是这个项目最被低估的资产**——`geo-ktv-changsha.yaml` 里的 agentPrompt、一模一策设计、执行拓扑包含了真正的业务智慧，但当前被存入数据库后再也无人读取。打通 agentPrompt → Aida 执行这一条路，ROI 最高。

3. **下一步只有一件事：造门**——系统已经有了仓库（DossierStore）、管理员（Aida）、监控室（Dashboard）、货物（mock 门店数据）、操作手册（blueprint agentPrompt），唯一缺的是让外面的 AI Agent 进来看货和下单的门。**MCP Server + Store Profile API = 门。**
