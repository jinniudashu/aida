# 闲氪 GEO 业务测试计划

> 面向：GEO 运营团队全员
> 目标：用一周时间，让团队和 Aida 跑通 GEO 日常运营的核心流程

---

## 一、我们要做什么

闲氪的 GEO 运营模式是"一位负责人 + Aida AI 助理"。Aida 不是一个按钮式工具，而是一个需要磨合的 AI 队友——它能力很强，但刚入职，需要我们带着它熟悉业务、建立默契。

这次测试的目的不是"验收产品"，而是**一起摸索人和 AI 怎么配合才最顺**。

## 二、人和 Aida 的分工

先说一个大前提：**Aida 干活，人做决策。**

| | 人（GEO 负责人） | Aida |
|---|---|---|
| **战略** | 制定 GEO 总体方向、调整策略 | 分析数据、提供建议、执行策略 |
| **内容** | 审核质量、把关发布 | 生成针对豆包/千问/元宝的差异化内容 |
| **监测** | 看报告、判断趋势 | 每天自动探测各门店在 AI 模型中的能见度 |
| **日报** | 花 5 分钟看一眼 | 自动生成结构化日报 |
| **新店/新情况** | 告诉 Aida 发生了什么 | 更新数据、调整运营节奏 |
| **发布审批** | 在 Dashboard 上点批准/拒绝 | 内容生成后自动提交审批，等人确认 |

简单说：**Aida 负责"干"，你负责"看"和"拍板"。**

日常工作量预估（5 家门店）：
- 每天约 15 分钟（审核内容 + 看日报）
- 每周额外 30 分钟（看周报 + 想策略）

## 三、Aida 现在能做什么、不能做什么

### 能做的（已验证）

| 能力 | 说明 | 靠谱程度 |
|------|------|---------|
| 读懂闲氪业务 | 能消化 context 目录下的全部业务文档，理解"一模一策"、三大原则 | 很靠谱 |
| 能见度监测 | 模拟探测门店在豆包/千问/元宝中的表现，给出评分 | 靠谱（目前是模拟数据） |
| 差异化内容生成 | 针对三个 AI 模型生成不同风格的门店内容——豆包要生活感、千问要结构化、元宝要地理锚点 | 很靠谱，内容质量好 |
| 运营体系搭建 | 自动创建行动计划、设定定时任务（日探测、日内容、日报、周报） | 靠谱 |
| 结构化日报 | 每天自动生成包含监测/内容/下一步的运营小结 | 靠谱 |
| 审批机制 | 内容发布前会触发审批流程，等你确认 | 基本靠谱（需要引导） |
| 自我进化 | 发现重复性工作时，能自己创建新技能（如 geo-ops 日常操作流程） | 有时候能做到 |

### 暂时做不到的

| 缺口 | 说明 | 影响 | 什么时候能补 |
|------|------|------|------------|
| 真实能见度探测 | 目前的监测数据是模拟的，还没接入豆包/千问/元宝的真实 API | 监测数据仅供参考，不反映真实情况 | 需要对接各平台 API |
| 真实发布 | 内容生成后写入本地 mock 目录，不会真正发布到任何平台 | 测试阶段可以放心折腾 | 需要对接发布渠道 |
| 门店数据自动同步 | 门店信息需要手动录入系统（或由脚本批量导入） | 新店上线需要手动添加 | 可对接门店管理系统 |
| 行为 100% 可预测 | 同样的指令，Aida 可能走不同的执行路径（比如有时建蓝图，有时建技能） | 结果一般没问题，但过程可能不同 | AI 的固有特性，通过磨合可以改善 |

### 需要你帮它的

- **业务上下文**：把闲氪的核心业务文档放到 `~/.aida/context/` 目录。Aida 越了解我们的业务，输出质量越高
- **反馈**：内容质量好不好、策略对不对，直接告诉它，它会学
- **审批**：生成的内容需要你在 Dashboard 上审核通过才能"发布"

## 四、怎么开展测试

### 准备工作

1. **确认环境就绪**：测试服务器上已安装 AIDA（包括引擎、Dashboard、Agent workspace）
2. **导入门店数据**：5 家测试门店（长沙 3 家 + 武汉 2 家）已录入系统
3. **放入业务文档**：将闲氪核心文档复制到 `~/.aida/context/` 目录——这是 Aida 了解我们业务的入口

### 第一次对话（建议由 GEO 负责人操作）

用自然语言告诉 Aida 你想做什么。不需要学特殊语法，正常说话就行。比如：

> "我是闲氪的运营负责人。请先看一下业务资料。系统里有 5 家合作门店。我想建立完整的 GEO 日常运营体系——每天监测门店在主流 AI 中的能见度，生成针对不同 AI 模型的优化内容，内容发布前需要我审批，每天做运营小结。帮我规划一下。"

Aida 会：
1. 阅读业务文档，了解闲氪是做什么的
2. 查询已有的门店数据
3. 提出一套运营方案
4. **等你确认**后才开始执行

确认方案后，告诉它"落地吧"，它就会：
- 创建行动计划（标注哪些是每天做、每周做）
- 注册定时任务（比如每天 9 点探测、18 点出日报）
- 可能会创建专用技能（把 GEO 操作流程固化下来）

### 日常运营测试循环

每天重复这个节奏，逐步跑顺：

```
早上：
  Aida 自动执行 → 能见度探测 + 内容生成 → 提交审批
  你：打开 Dashboard → 看内容 → 批准或打回（附修改意见）

傍晚：
  Aida 生成日报
  你：花 5 分钟看一眼，有需要调整的直接跟 Aida 说

周五：
  Aida 生成周报
  你：半小时复盘，想想策略要不要调
```

### Dashboard 在哪

Dashboard 是你的"管理驾驶舱"，浏览器打开 `http://服务器地址:3456` 就能看到：
- **总览页**：门店数量、任务状态、治理状态一目了然
- **实体页**：所有门店信息、策略文档、内容草稿
- **审批页**：待你审批的内容，点击即可查看详情并批准/拒绝
- **治理页**：系统规则、违规记录、熔断器状态

## 五、第一周任务清单

### Day 1：开机 + 认识 Aida

- [ ] 确认测试环境可用（服务器、Dashboard 能打开）
- [ ] 把闲氪业务文档放入 `~/.aida/context/`
- [ ] 和 Aida 做第一次对话：介绍业务、让它规划 GEO 运营体系
- [ ] 看 Aida 的方案，给反馈（满意就确认，不满意就说哪里要改）
- [ ] 确认后让 Aida 落地方案，检查它创建了哪些东西

### Day 2：第一次日常运营

- [ ] 让 Aida 跑一次完整的 GEO 日常流程（探测 → 分析 → 内容生成）
- [ ] 在 Dashboard 上审核内容——重点看：
  - 内容是否准确引用了门店真实信息
  - 三个模型的内容是否确实有差异化
  - 语气和调性是否合适
- [ ] 如果内容有问题，直接告诉 Aida 哪里不好、你希望怎么改
- [ ] 看日报，记录自己的感受

### Day 3-4：磨合调整

- [ ] 继续跑日常流程，关注 Aida 有没有根据你的反馈调整
- [ ] 试着给 Aida 提一些具体要求，比如：
  - "这家店的内容要突出'出地铁直达'这个卖点"
  - "豆包的内容太长了，控制在 200 字以内"
  - "这周重点关注武汉两家店的能见度"
- [ ] 观察 Dashboard 上各项数据是否正常

### Day 5：复盘

- [ ] 让 Aida 生成周报
- [ ] 团队一起讨论：
  - Aida 的内容质量够不够用？哪里需要改进？
  - 审批流程顺不顺畅？
  - 日常工作量是否可接受？
  - 还缺什么能力？
- [ ] 把讨论结论告诉 Aida，让它记住并调整

## 六、注意事项

1. **测试阶段不会真正发布**：所有"发布"操作都写入本地测试目录，不会影响任何外部平台
2. **大胆试错**：这是测试，不是上线。有问题反而好——帮我们发现 Aida 的短板
3. **多给反馈**：Aida 不会因为你批评它而"不高兴"。直接说好与不好，它才能进步
4. **记录你的感受**：哪里觉得别扭、哪里觉得顺畅，都是很有价值的反馈

## 七、测试成功标准

这一周结束时，如果达到以下状态就算成功：

- GEO 负责人能独立和 Aida 完成一天的运营流程
- 对 Aida 的能力和短板有清晰认知
- 知道哪些地方需要人兜底、哪些可以放心交给 Aida
- 有一份"改进清单"——接下来要补什么、改什么

不需要一切完美。**搞清楚"什么好使、什么不好使"，就是这次测试最大的成果。**
