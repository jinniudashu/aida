# BPS 引擎价值反思与架构瘦身建议

> 日期：2026-03-03
> 背景：全面审视 AIDA 项目各部分的实用价值，回答四个根本性问题

---

## 问题 1：Agent 能写代码、Skill 能固化流程，BPS 的价值在哪里？

**结论：BPS 的大部分价值正在被 Agent 能力的进化所吞噬。**

| BPS 提供的能力 | Agent + Skill 已经能做的 |
|---|---|
| 声明式 YAML 描述业务流程 | Agent 直接写代码/Skill 描述流程 |
| 7-state 状态机管控进程生命周期 | Agent session 已有生命周期管理 |
| 规则引擎（事件→指令映射） | Agent 本身就是一个"活的规则引擎"，天然处理事件→行动 |
| 非确定性事件路由给 LLM 判断 | 这正是 Agent 的本职工作 |
| Dossier 版本化存储 | 任何数据库/文件系统 + Agent 代码即可实现 |
| 进程可视化 (Dashboard) | 有独立价值，但不依赖 BPS 引擎 |

**BPS 仅剩的真正差异化价值：**

1. **可审计性** — 结构化的状态转换日志、context snapshot。但用简单的 audit log 中间件也能实现。
2. **非技术人员可读性** — YAML 蓝图理论上比代码更易读。但 BPS 的 YAML 格式（services + events + instructions + rules 四段式）对非技术人员并不比 Python 代码更友好。写一个蓝图需要理解 6 元组、9 种 SysCall、7 种进程状态——这本身就是一门编程语言。
3. **约束与治理** — 状态机强制合法转换，防止混乱。但 Agent 本身就可以在 Skill 中实现同样的约束。

**根本性的矛盾：** BPS 试图用"声明式规范"来约束 Agent 行为，但 Agent 的核心优势恰恰是灵活应变。用一个刚性框架去管理一个柔性智能体——产生了架构上的阻抗不匹配。

## 问题 2：BPS 是否只需作为 tools 注册，而不必写进 SOUL/AGENTS？

**结论：是的，绝对是。当前做法有严重的资源浪费。**

现状问题：

- BPS-Expert SOUL.md (269行) 完整复述 BPS 六元组理论、状态机、YAML 格式规范——每次对话都占用 Agent 的 context window
- Aida AGENTS.md (254行) 大量篇幅描述 BPS 相关操作规程、协作协议
- 结晶化框架 (7 场景) 是关于"何时使用 BPS"的元决策框架——一个工具需要一个决策框架来决定何时使用它，说明它不够自然

如果 BPS 只是作为 9 个 tools 注册：

- tool description 本身就说明了用途和参数
- Agent 按需调用，不需要把整个理论体系塞进 SOUL
- 节省大量 context window 给真正的业务推理
- Aida 的人格和智慧应该体现在对业务的理解上，而不是对 BPS 规范的熟练程度上

**本质问题：** 当前做法是在用"教育 Agent 学习 BPS"来替代"让 BPS 对 Agent 足够直觉化"。如果一个工具需要 500+ 行的说明书才能让 Agent 正确使用，说明工具的抽象层次有问题。

## 问题 3：业务流程固化用 BPS 还是 Skill？

**结论：Skill 是更自然的选择。**

| 维度 | BPS Blueprint | OpenClaw Skill |
|---|---|---|
| 表达形式 | YAML 声明式（需要学习专用 DSL） | 代码/配置（Agent 天然理解） |
| 执行方式 | 需要 BPS 引擎运行时 | OpenClaw 原生执行 |
| 灵活性 | 受限于 9 种 SysCall | 任意代码逻辑 |
| 可组合性 | composite service + rules 链 | Skill 可调用其他 Skill/Tool |
| 状态管理 | 内置 7-state 状态机 | 需要自行实现（但通常更简单） |
| 可审计性 | 内置 snapshot + event log | 需要自行添加 |
| 修改门槛 | 编辑 YAML + 理解规则拓扑 | 修改代码/配置 |
| 对 Agent 透明度 | Agent 是被调度的执行者 | Agent 直接拥有能力 |

**建议：** 大部分业务流程固化应该用 Skill。只有满足以下**全部**条件时才值得用 BPS：

- 需要严格的多方协作审批链（不能跳步）
- 需要完整的状态审计和回溯能力
- 流程涉及合规/法务要求，需要证明"每一步都按规定执行了"

这些场景在大多数 AI-native 组织中是少数。ADR 文档自己也说了"只有 20% 需要结晶化"——但为了这 20%，建造了一个完整的虚拟机。

## 问题 4：各部分实用价值全面评估

### 高价值（值得保留和发展）

| 成果 | 价值类型 | 理由 |
|---|---|---|
| **BPS 理论规范文档** | 学术/思想 | "组织运营即图灵计算"是有深度的洞见，值得发表。但应是论文，不是运行时引擎 |
| **Dossier 版本化存储** | 工程 | 版本化 JSON 文档 + 审计轨迹是通用有用的模式，可独立于 BPS 引擎使用 |
| **Dashboard 可视化** | 产品 | 进程拓扑图 + 实时状态 + ATDD 测试循环，无论底层引擎是什么都有用 |
| **Agent workspace 设计模式** | 架构 | IDENTITY/SOUL/AGENTS 三文件分工清晰，是好的 Agent 定义实践 |
| **~/.aida/ 数据分离** | 工程 | 代码与业务数据分离，loadAidaProject() 一键初始化 |
| **OpenClaw 技术研究报告** | 知识资产 | 对 OpenClaw 架构的深度分析，有持续参考价值 |

### 价值可疑（投入产出比待商榷）

| 成果 | 问题 | 建议 |
|---|---|---|
| **bps-engine 完整引擎 (209 tests)** | 完整的业务虚拟机，但只跑过 GEO KTV 一个测试蓝图，没有真实 Agent 使用过 | 精简为轻量级 "process tracker + audit log" |
| **BKM 知识管理 (5层+6域)** | 5 层 × 6 域 = 30 种组合，实际可能只需 "全局" + "项目" 两层 | 简化为 key-value 知识库 + 简单继承 |
| **sys:project-init 系统蓝图** | 一次性初始化形式化为 8 services + 8 rules。setup wizard 脚本就能做 | 用普通代码/Skill 实现 |
| **结晶化判断框架 (7 场景)** | 关于"何时使用 BPS"的元决策框架。BPS 价值存疑则框架悬空 | 如简化 BPS 角色，此框架可大幅精简 |
| **SBMP 5步建模法** | 学术化建模方法论，没有真实业务验证 | 保留为文档参考，不嵌入 Agent context |

### 低价值（可以归档）

| 成果 | 理由 |
|---|---|
| **erpsys (Django 版)** | 已定义为"仅供借鉴"，使命已完成 |
| **BPS-Expert Agent** | 如 BPS 精简，专门"写 YAML 蓝图"的 Agent 不再需要。Aida 直接调用工具或写 Skill |
| **Org-Architect Agent** | 仅有框架性定义 (SOUL.md 41 行)，没有实际运行过。Agent 管理可以是 Aida 的一项能力 |
| **规则引擎双模态** | 确定性规则太简单（if-else），不值得引擎；非确定性就是 LLM 调用，不需要引擎包装 |

## 核心建议：BPS 从"运行时引擎"退化为"描述规范 + 轻量工具"

```
当前架构：
  Agent → BPS Engine (full VM) → 执行业务流程

建议架构：
  Agent → Skill/Code (直接执行) + 轻量审计层 (process tracker)
         ↓
       BPS 规范作为"设计时参考"（帮助 Agent 理解和分析业务流程）
       而非"运行时引擎"（不强制所有操作经过 BPS VM）
```

具体动作：

1. **保留** BPS 规范文档作为业务建模的思维框架（学术价值）
2. **保留** Dossier 存储 + Dashboard 作为独立可用的组件
3. **精简** bps-engine 为 "process tracker"——记录谁在做什么、进展到哪步、结果是什么
4. **删除** 规则引擎的运行时执行——让 Agent 自己决定下一步做什么
5. **让 Skill 成为流程固化的主要手段**，BPS 工具作为"观测和审计"辅助
6. **大幅缩减** Agent workspace 中的 BPS 相关内容，让 Aida 回归"智能助理"本色

**AIDA 项目真正的核心资产不是 BPS 引擎，而是"AI-native 组织运营"这个思想框架和 Agent 协作模式。引擎只是载体，不应该让载体的复杂度遮蔽了思想的价值。**
