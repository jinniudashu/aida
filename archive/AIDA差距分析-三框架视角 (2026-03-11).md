# AIDA 差距分析：操作系统 × 控制论 × 有限理性

> 以三个计算机科学/认知科学经典框架重新审视 AIDA 的架构差距
> 日期：2026-03-11（修订版 v1.1：新增"管理层 vs 治理层"架构命名章节）

---

## 分析方法

前序评审报告（`archive/市场即计算-差距分析评审报告 (2026-03-11).md`）指出"市场即计算"框架的域边界问题——市场类比解决的是去中心化协调，而 AIDA 面对的是单 Agent 中心化执行。评审报告提出三个更匹配的替代框架：

| 框架 | 创始人 | 核心问题 | 对 AIDA 的适用性 |
|------|--------|---------|-----------------|
| **操作系统理论** | Dijkstra, Thompson | 如何安全高效地管理共享资源？ | AIDA 就是一个 Agent 运行时——进程、内存、调度、保护 |
| **控制论** | Wiener, Ashby | 系统如何通过反馈维持稳定并趋向目标？ | AIDA 的管理层是内部控制器，Agent 是被控对象 |
| **有限理性** | Herbert Simon | 受限主体如何做出"足够好"的决策？ | Agent (LLM) 的上下文窗口和推理能力有限，AIDA 是其决策环境 |

每个框架产生一组独立的差距发现。三组发现交叉验证后，按工程优先级统一排序。

所有断言均通过源码交叉验证（见各节证据引用）。

---

## 框架一：操作系统视角

**核心类比**：AIDA 是 Agent 的操作系统——Entity 是内存，Task 是进程，Governance 是保护环，bps_scan_work 是调度器。

### OS-1：进程管理

**AIDA 已有**：
- 5 状态机：OPEN → IN_PROGRESS → COMPLETED / FAILED / BLOCKED（`src/engine/state-machine.ts`）
- 进程表：SQLite `bps_processes`，含 UUID + 数字 PID + parentId + previousId
- 进程树重建：`getProcessTree()` 支持 50 层深度（`src/store/process-store.ts`）
- 审计日志：task action log（created/state_changed/completed/failed + timestamp + details）
- 上下文快照：`bps_context_snapshots` 表，SHA256 去重

**差距**：

| 差距 | OS 类比 | 严重度 | 证据 |
|------|---------|--------|------|
| **无进程组** | 无法 `kill -TERM -pgid`：不能批量操作同一 Action Plan 下的全部任务 | 高 | ProcessDef 无 groupId 字段 |
| **无进程间信号** | 无 SIGTERM/SIGSTOP：父任务无法通知子任务"停止"或"加速" | 高 | EventEmitter 是广播模型，无定向投递 |
| **无僵尸回收** | COMPLETED/FAILED 进程永不清理，无保留策略 | 中 | `process-store.ts` 无 archive/delete 方法 |
| **无等待语义** | 无 waitpid()：父任务完成后，子任务不会自动收到通知 | 中 | 父子关系仅为 parentId 字段，无阻塞等待 |
| **无资源限额** | 无 ulimit：单个 Agent 可无限创建任务，无上限 | 中 | `bps_create_task` 无配额检查 |

### OS-2：内存管理

**AIDA 已有**：
- 版本化文档存储：Dossier 的 copy-on-write 语义（每次 commit 创建新版本）
- 全局寻址：erpsysId UUID + (entityType, entityId) 元组
- smartMerge：对象浅合并 + 数组追加（`dossier-store.ts:363-376`）
- 不可变审计轨迹：所有版本永久保留

**差距**：

| 差距 | OS 类比 | 严重度 | 证据 |
|------|---------|--------|------|
| **无写保护** | 无 mprotect()：任何 Agent 可写任何实体，Governance 是后置检查不是前置屏障 | 高 | `action-gate.ts` 在工具层拦截，非内存层 |
| **无并发锁** | 无 mutex/semaphore：两个 Agent 同时更新同一实体可能数据竞争 | 高 | SQLite 行锁是底层保障，BPS 层无锁抽象 |
| **无实体关系** | 无指针/引用：实体间无外键、无依赖声明、无关联图 | 高 | `src/schema/` 无关系定义 |
| **无 GC 策略** | 无页面回收：Dossier 版本无限累积 | 低 | 设计意图（审计合规），非缺陷 |

### OS-3：调度器

**AIDA 已有**：
- 优先级排序：`ORDER BY priority DESC, pid ASC`（`process-store.ts:254`）
- 全景扫描：`bps_scan_work` 一次返回全部活跃任务

**差距**：

| 差距 | OS 类比 | 严重度 | 证据 |
|------|---------|--------|------|
| **Priority 不可设** | nice 值存在但 shell 未暴露：`bps_create_task` 不接受 priority 参数 | **关键** | `tools.ts:74-81` 参数列表无 priority |
| **无抢占** | 无时间片：IN_PROGRESS 任务永不被中断，可无限占用 | 高 | 无 timeout 或 watchdog 机制 |
| **无老化** | 无 aging：低优先级任务可能永久饥饿 | 中 | 无 MLFQ 或优先级提升逻辑 |
| **无 deadline 调度** | 无 EDF（Earliest Deadline First）：任务无截止时间字段 | 高 | ProcessDef 无 deadline 字段 |
| **无调度策略声明** | 策略隐含在 Agent 的 LLM 推理中，引擎不保证一致性 | 中 | 无 SchedulerPolicy 配置 |

### OS-4：保护环

**AIDA 已有**：
- 二环模型：Ring 3（Agent 工具调用）→ Ring 0（ActionGate 治理检查）
- 5 个受保护系统调用：update_entity, create_task, update_task, complete_task, create_skill
- 三级裁决：PASS / REQUIRE_APPROVAL / BLOCK
- 熔断器：NORMAL → WARNING → RESTRICTED → DISCONNECTED

**差距**：

| 差距 | OS 类比 | 严重度 | 证据 |
|------|---------|--------|------|
| **治理绕过** | Ring 3 程序绕过 MMU 直接写物理内存 | **关键** | Agent 的 write 工具不经 ActionGate；`bps_load_blueprint`（tools.ts:573）和 `bps_register_agent`（tools.ts:759-777）也未纳入 WRITE_TOOLS |
| **无用户隔离** | 无 UID/GID：所有 Agent 以"root"运行 | 高 | ConstraintScope 无 users 字段 |
| **无能力撤销** | 无运行时 revoke：授权后只能重启 Agent 才能收回 | 中 | 约束仅在加载时绑定 |
| **失败封闭风险** | 约束表达式错误导致工具全部 BLOCK | 中 | expr-eval 求值异常 → 默认 BLOCK |

### OS-5：进程间通信

**AIDA 已有**：
- EventEmitter 广播：5 个进程事件 + 4 个治理事件 → Dashboard SSE 转发
- 共享数据库：单一 SQLite 实例作为隐式共享内存
- Dossier 作为结果传递：任务完成时 auto-commit 到实体

**差距**：

| 差距 | OS 类比 | 严重度 | 证据 |
|------|---------|--------|------|
| **无点对点消息** | 无管道/socket：EventEmitter 是广播，无定向投递 | 高 | 无消息队列或信箱抽象 |
| **无阻塞 RPC** | 无 call/wait：创建子任务后必须轮询完成状态 | 中 | `bps_create_task` 是 fire-and-forget |

## 框架二：控制论视角

**核心类比**：AIDA 是一个**控制系统**——Governance 是控制器，Agent 行为是被控对象，Entity 状态是被控变量。

### CTL-1：反馈回路清单

对 AIDA 全部控制路径进行了穷举式梳理：

#### 闭环（自动）

| 回路 | 感知 | 控制动作 | 阻尼 | 稳定性 |
|------|------|---------|------|--------|
| **违规→熔断器** | 违规计数（1h 窗口） | 状态升级（NORMAL→WARNING→RESTRICTED→DISCONNECTED） | 窗口滑动衰减 | **单向稳定**：只能升级，不能自动降级 |
| **Dossier 版本递增** | commit() 触发 | 版本号 +1 | 无（仅追加） | 单调递增，不可能振荡 |
| **Agent 会话结束→任务状态** | OpenClaw subagent.ended(outcome) | outcome='ok'→complete, outcome='error'→fail | 无重试 | 即时、无振荡、但无恢复 |

#### 开环（需人工/Agent 介入）

| 回路 | 感知 | 控制动作 | 触发方 |
|------|------|---------|--------|
| **审批工作流** | REQUIRE_APPROVAL 裁决 | Dashboard 人工批准/拒绝 | 人类 |
| **熔断器重置** | DISCONNECTED 状态 | `resetCircuitBreaker()` | 人类 |
| **治理规则更新** | 违规模式观察 | 修改 governance.yaml + `bps_load_governance` | 人类 |
| **任务恢复** | `bps_scan_work` 扫描失败 | Agent 决定重试/升级/放弃 | Agent |
| **规则评估** | `bps_next_steps` 查询 | Agent 决定执行哪条规则 | Agent |
| **Heartbeat 调度** | Cron 触发 | Agent 选择工作项执行 | Agent |

**开环比例：~80%** — AIDA 的绝大多数纠正性操作需要人类或 Agent 主动介入。

### CTL-2：熔断器深度分析

熔断器是 AIDA 唯一的全自动闭环控制器。逐项分析其控制论性质：

**状态转移规则**（`action-gate.ts:271-303`）：
```
1 CRITICAL 违规 / 1h  → DISCONNECTED（全面封锁，需人工重置）
3+ HIGH 违规 / 1h     → RESTRICTED（全面封锁，需人工重置）
2 HIGH 违规 / 1h      → WARNING（写操作需审批）
0 违规 / 1h           → NORMAL（正常放行）
```

**控制论分析**：

| 性质 | 评估 | 说明 |
|------|------|------|
| **单调安全性** | 优秀 | 一旦 DISCONNECTED，Agent 无法自行恢复——这是安全系统的正确设计 |
| **振荡可能性** | 低 | 窗口滑动衰减防止短周期振荡。但存在**长周期振荡风险**：人工重置 → Agent 立即触发新违规 → 再次 DISCONNECTED → 再次重置……当前无振荡检测 |
| **死锁风险** | 存在 | DISCONNECTED 状态下 Agent 无法执行写操作；如果人类长时间不在，系统完全停摆 |
| **自动恢复** | **缺失** | `cooldown` 字段已定义（`action-gate.ts:28`）但**从未使用**。config 中有 cooldown 配置位但代码未实现超时自动恢复 |
| **增益控制** | 无 | 违规→状态转移是固定阈值，无比例/积分/微分调节 |

**关键发现：cooldown 是幽灵代码。** ActionGate 的配置接口中定义了 `cooldown` 字段（用于自动恢复超时），但 `updateCircuitBreaker()` 方法中没有使用它。这意味着设计者**曾经考虑过自动恢复，但未实现**。

### CTL-3：必要多样性分析（Ashby 定律）

> *"只有多样性能消灭多样性。"* ——Ashby, 1956

控制器（Governance）的控制多样性必须 ≥ 被控系统（Agent 行为）的状态多样性。

**被控系统多样性**：
- 5 个写操作工具 × 不确定数量的 entityType × 不确定数量的 entityId × 任意 data 字段组合
- 保守估计：10³ ~ 10⁴ 种不同的写操作上下文

**控制器多样性**：
- 3 种裁决（PASS / REQUIRE_APPROVAL / BLOCK）
- 4 种熔断器状态
- 约束匹配维度：tool × entityType × dataField（~3 维度）
- 实际控制选项：约 30-50 种不同的约束配置

**Ashby 缺口**：控制器多样性（~50）远小于被控系统多样性（~10⁴）。

**实际影响**：
- Governance 足以**保障安全**（阻止已知的坏操作）
- Governance **不足以保障活性**（确保系统朝正确方向前进）
- Governance **无法保障最优**（从多个合法操作中选择最好的）

**类比**：一个只有"红灯"和"绿灯"的交通系统可以防止碰撞（安全），但不能优化通行效率（活性）——这需要左转信号、潮汐车道、智能感应等更多控制维度。

### CTL-4：缺失的控制路径

以下是 AIDA 中**应该存在但不存在的反馈回路**：

| 缺失回路 | 感知（已有） | 控制（缺失） | 影响 |
|----------|------------|-------------|------|
| **任务完成→规则自动评估** | `task:completed` 事件已 emit | 不会自动查询 `bps_next_steps` 并创建下游任务 | Blueprint 规则永远不会被自动执行——Agent 必须手动查询并手动创建 |
| **违规模式→约束自适应** | 违规记录已持久化 | 不反馈到约束定义 | 同一类违规发生 100 次，约束仍然不变 |
| **审批通过率→策略建议** | 审批记录已持久化 | 不聚合分析 | 某约束 95% 被批准，系统不会建议放宽 |
| **任务延迟→升级** | createdAt 已记录 | 无"超时"检测，无自动升级 | 任务 OPEN 三天无人处理，系统无感知 |
| **实体变更→依赖通知** | `dossier:committed` 事件已 emit | 不通知依赖实体的相关任务 | "店铺地址变了"不会自动提醒正在处理该店铺的任务 |
| **Skill 使用→效果评估** | Skill 文件已创建 | 无调用计数、成功率、平均耗时 | 好 Skill 和坏 Skill 无法区分 |

### CTL-5：控制架构总体特征

**AIDA 的控制论肖像**：

1. **约束执行偏向**：强于阻止坏事（安全），弱于推动好事（活性）
2. **反应式，非处方式**：在违规发生后响应，不主动引导 Agent 走向最优状态
3. **人工依赖恢复**：熔断器重置、约束调整、策略更新全部需要人类介入
4. **事件驱动感知，开环执行**：通过 SSE 实时感知状态变化，但纠正动作是 Agent/人类发起的
5. **安全充分，活性不足**：Ashby 定律下控制器多样性仅够保障安全性

## 框架三：有限理性视角

**核心类比**：Agent (LLM) 是一个有限理性决策者——上下文窗口有限、推理容易出错、容易"说而不做"。AIDA 应该是帮助 Agent 做出"足够好"决策的**决策环境**。

### BR-1：信息呈现

**AIDA 已有**：
- `bps_scan_work`：一次调用返回全工作景观（失败/开放/进行中/已完成任务 + 活跃计划）——这是最符合有限理性的工具
- `bps_query_entities`：默认限制 50 条，按 updated_at DESC 排序
- `bps_next_steps`：返回下游规则拓扑

**差距**：

| 差距 | Simon 的术语 | 严重度 | 证据 |
|------|-------------|--------|------|
| **无排序/排名** | 缺少**注意力过滤** | **关键** | 所有查询工具返回原始数据，无"按紧急度排序"或"按业务价值排序"；`bps_scan_work` 内各分组无排序（`tools.ts:444-465`） |
| **无信息摘要** | 缺少**抽象层次** | 高 | 50 个实体以完整 JSON 返回，无"3 个需要注意的实体 + 摘要"模式 |
| **无分页提示** | 缺少**搜索终止信号** | 中 | 工具不返回 `{hasMore: true, nextOffset: 50}`——Agent 不知道是否遗漏了重要数据 |
| **无关联高亮** | 缺少**上下文聚焦** | 中 | 查询实体时不标记"此实体有 3 个关联任务正在进行" |

**R4-R7 基准测试中的表现**：多个模型反复出现"说而不做"模式——用高质量的自然语言描述完美计划，但不调用工具执行。这恰恰是有限理性理论预测的行为：**当决策环境不提供"已够好，现在执行"的信号时，受限主体倾向于停留在信息搜集阶段**（Simon 称之为"搜索过度"）。

### BR-2：决策支持

**AIDA 已有**：
- 治理层：PASS/BLOCK/REQUIRE_APPROVAL 三级裁决——这是**约束型决策支持**（告诉 Agent "不能做什么"）
- AGENTS.md 二层路由：信号→层级分类决策表——这是**元决策支持**（告诉 Agent "如何分类问题"）
- TOOLS.md 工具分组：读/写分类 + 常用模式——这是**认知脚手架**（告诉 Agent "工具怎么组合"）

**差距**：

| 差距 | Simon 的术语 | 严重度 | 证据 |
|------|-------------|--------|------|
| **无正向推荐** | 缺少**启发式** | **关键** | 没有工具返回"建议下一步做 X，因为 Y 是瓶颈"；`bps_next_steps` 返回规则拓扑数据，不是建议 |
| **无"该执行了"信号** | 缺少**满意化阈值** | **关键** | 无机制告诉 Agent"你已经收集了足够信息，现在执行"——这直接解释了"说而不做"反模式 |
| **无置信度信号** | 缺少**不确定性量化** | 高 | 工具返回 `{success: true/false}`，无 `{confidence: 0.73}` |
| **无成本信号** | 缺少**资源感知** | 高 | Agent 不知道"创建一个实体"vs "写一篇 2000 字文章"的成本差异 |

### BR-3：满意化机制

Simon 的有限理性核心：决策者不追求最优，追求"足够好"（satisficing）。这需要**质量阈值**——"做到这个程度就可以了"。

**AIDA 完全缺少满意化机制**：

- 任务状态是二值的：COMPLETED 或非 COMPLETED。无 PARTIAL_SUCCESS、GOOD_ENOUGH 等中间状态
- `bps_complete_task` 的 `result` 参数是非结构化的 `Type.Unknown()`——无法查询"哪些任务只完成了 70%"
- Action Plan 的 `resources` 字段（预算约束）存在于 Skill 文档中，但**无工具检查预算剩余**
- 无"收益递减"信号——Agent 不知道继续优化一个实体是否还有边际价值

**后果**：Agent 在两个极端之间摇摆——要么"说而不做"（过度搜索），要么"做了就完"（无质量评估）。中间的"做到足够好然后停止"没有引擎级支持。

### BR-4：注意力管理

**AIDA 已有**：
- Heartbeat 周期：定时唤醒 Agent 聚焦于有界工作集（失败/到期/模式）
- BOOT.md 启动序列：4 步引导 Agent 加载最关键的上下文
- Memory 系统：MEMORY.md + daily notes 实现跨会话持续性

**差距**：

| 差距 | Simon 的术语 | 严重度 | 证据 |
|------|-------------|--------|------|
| **无信息过载防护** | 缺少**注意力容量管理** | 高 | 当实体数量从 10 增长到 100 再到 1000 时，`bps_scan_work` 的返回量线性增长，无摘要/聚合 |
| **无上下文溢出预警** | 缺少**认知负荷感知** | 中 | 工具不知道当前会话已消耗了多少 token，无法建议"这个查询可以推迟到下次 heartbeat" |
| **Heartbeat 仅有时间边界** | 缺少**信息边界** | 中 | Heartbeat 定义了"何时关注"但不定义"关注多少"——Step 1 "Scan failures" 可能返回 200 个失败任务 |

### BR-5：选择架构

**AIDA 已有**：
- 16 个 BPS 工具，TOOLS.md 分为读（9）/ 写（5+1）两组
- 常用模式指引（Status check, Entity lifecycle, Blueprint load）
- 二层路由决策表

**差距**：

| 差距 | Simon 的术语 | 严重度 | 证据 |
|------|-------------|--------|------|
| **工具扁平呈现** | 缺少**选择层次** | 中 | 16 个工具以平铺数组返回（`tools.ts:849-882`），无元数据标注 `category` 或 `frequency` |
| **无使用场景引导** | 缺少**情境嵌入** | 中 | Agent 需阅读 TOOLS.md 才能知道工具组合模式；工具自身不嵌入"通常在 X 之后调用" |
| **治理失败消息不透明** | **认知负荷过高** | 中 | BLOCK 和工具逻辑错误抛出相同的 Error 类型——Agent 需要解析错误消息文本才能区分"治理拦截"vs"参数错误" |

### BR-6：搜索 vs 执行权衡

**这是 AIDA 最关键的有限理性差距。**

基准测试 R4-R7 反复验证的"说而不做"模式本质上是一个**搜索-执行决策点的缺失**：

1. Agent 接收到目标（如"为三家店铺做一模一策的 GEO 运营"）
2. Agent 开始搜索（查询实体、查询任务、查询服务……）
3. Agent 积累了大量上下文——此时应该**停止搜索，开始执行**
4. 但没有任何信号告诉 Agent"你已经知道得够多了"
5. Agent 继续搜索（或更常见的：在输出文本中**模拟执行**——用自然语言描述它会做什么，但不实际调用工具）

**缺失的机制**：
- 无"搜索饱和"信号（"你已查询了所有相关实体，没有更多了"）
- 无"信息增量递减"检测（"最近 3 次查询没有返回新信息"）
- 无"执行提醒"触发（"你已分析了 5 分钟，请调用至少一个写操作工具"）

## 优先级路线图

基于三框架交叉验证的严重度和跨框架确认数，排列工程改造优先级：

### P0-a：治理绕过封堵（OS-保护 × CTL-控制器盲区）

**问题**：`bps_load_blueprint`、`bps_register_agent` 和 Agent 的 `write` 工具绕过 ActionGate。
**方案**：
1. 将 `bps_load_blueprint` 和 `bps_register_agent` 加入 `WRITE_TOOLS` 集合——**一行代码变更**
2. 在 `wrapWithGovernance` 中为 `bps_register_agent` 定义专用约束 scope（工具名匹配）
3. OpenClaw `tools.exec.security: "allowlist"` 中限制 `write` 工具的目标路径
**验收**：下一轮 E2E 测试中 0 次文件 I/O 治理绕过
**工程量**：小（1-2 天）

### P0-b：调度器基础修复（OS-调度器 × BR-排序 × CTL-超时）

**问题**：Priority 不可设、无 deadline、任务无超时。
**方案**：
1. `bps_create_task` 暴露 `priority` 和 `deadline` 参数
2. `bps_scan_work` 返回结果按 `deadline ASC, priority DESC` 排序
3. `bps_scan_work` 新增 `overdue` 分组：deadline 已过但未完成的任务
4. ProcessDef schema 新增 `deadline: Type.Optional(Type.String({ format: 'date-time' }))`
**验收**：Agent 可创建带截止时间的任务，scan_work 优先返回逾期任务
**工程量**：小（2-3 天）

### P0-c：outcome 结构化（CTL-反馈 × BR-满意化）

**问题**：任务完成后无结构化结果评估，无法聚合统计。
**方案**：
1. `bps_complete_task` 的 `result` 参数增加可选的结构化格式：`{ outcome: 'success' | 'partial' | 'failed', quality?: number, notes?: string }`
2. `bps_scan_work` 新增 "最近完成任务的 outcome 分布"摘要（如：success 80%, partial 15%, failed 5%）
3. Dashboard Overview 页新增 outcome 分布图
**验收**：`bps_scan_work` 返回含 outcome 统计的摘要行
**工程量**：中（3-4 天）

### P1-a：熔断器自动恢复（CTL-稳定性）

**问题**：`cooldown` 字段已定义但未实现——DISCONNECTED 状态需人工重置，存在死锁风险。
**方案**：
1. 实现 `cooldown` 逻辑：`updateCircuitBreaker()` 检查 `lastStateChangeTime + cooldownMs`，若已过冷却期且窗口内无新违规，自动降级一档（DISCONNECTED→RESTRICTED→WARNING→NORMAL）
2. 添加振荡检测：若 1h 内状态转移 > 3 次，emit `governance:oscillation_detected` 事件并锁定当前状态
3. Dashboard 治理页展示"下次自动恢复时间"倒计时
**验收**：熔断器在冷却期后自动降级；振荡时锁定而非反复跳变
**工程量**：中（2-3 天）

### P1-b：信息摘要层（BR-信息架构）

**问题**：所有查询工具返回原始数据，Agent 上下文被低价值信息淹没。
**方案**：
1. `bps_scan_work` 增强：
   - 各分组只返回 top-N（按 deadline ASC, priority DESC），附 `{total: M, showing: N}` 元数据
   - 新增 `summary` 字段："3 overdue tasks, 12 open, 2 failed since last scan"
2. `bps_query_entities` 增加 `brief` 模式：只返回 entityType + entityId + 最近修改摘要，不返回完整 currentValues
3. `bps_next_steps` 增加 `recommendation` 字段："Recommended: start service X (highest priority downstream)"
**验收**：`bps_scan_work` 的 brief 模式响应 token 数 < 完整模式的 30%
**工程量**：中（3-5 天）

### P1-c：Skill 使用追踪（CTL-反馈 × OS-进程统计）

**问题**：Skill 创建后无调用计数、成功率、最后使用时间。
**方案**：
1. 新增 SQLite 表 `bps_skill_metrics`（skillName, invokedAt, outcome, durationMs）
2. ProcessTracker 在 completeTask/failTask 时，若关联 serviceId 可映射到 Skill，自动记录
3. `bps_scan_work` 新增 "dormant skills" 分组：90 天未使用的 Skill 列表
4. Dashboard 新增 Skill 排行榜面板
**验收**：Skill 使用统计可查询，90 天未用 Skill 被标记
**工程量**：中（3-4 天）

### P2-a：进程组（OS-进程管理）

**问题**：无法批量管理同一 Action Plan 下的全部任务。
**方案**：
1. ProcessDef 新增 `groupId` 字段（可选，默认为空）
2. `bps_create_task` 暴露 `groupId` 参数
3. 新增 `bps_batch_update` 工具：按 groupId 批量更新任务状态（如全部取消）
**验收**：可一次调用取消某 Action Plan 下的全部 OPEN 任务
**工程量**：中（2-3 天）

### P2-b：实体关系声明（OS-内存引用 × CTL-变更传播 × BR-关联高亮）

**问题**：三框架均独立识别——实体间无关系，是 AIDA 最深层的结构性差距。
**方案**（第一步，仅声明式）：
1. Dossier 新增可选 `relations` 字段：`Array<{ targetEntityType, targetEntityId, relationType: 'depends_on' | 'part_of' | 'references' }>`
2. `bps_update_entity` 接受 `relations` 参数
3. `bps_get_entity` 返回关联实体的摘要（类型 + ID + 最近修改时间）
4. **不做级联更新**——变更传播由 Agent 决定（符合"Agent 自治"设计哲学）
**验收**：实体可声明引用关系，查询时可见关联实体
**工程量**：中-大（4-6 天）

### P3：审批模式学习（CTL-策略优化）

**问题**：审批记录仅用于审计，不反馈到策略建议。
**方案**：
1. Dashboard 治理页新增"约束效果分析"：按 constraintId 聚合（拦截次数 / 通过率 / 拒绝率）
2. 当某约束审批通过率 > 90%（且样本 > 20），显示建议："此约束可能过于严格，考虑放宽"
3. 当某约束审批拒绝率 > 80%，显示建议："此约束有效，考虑升级为 BLOCK"
4. **不做自动调整**——策略变更始终由人类确认
**验收**：Dashboard 展示约束效果统计和策略建议
**工程量**：中（3-4 天）

## 架构命名修订：管理层 vs 治理层

### 问题

AIDA 当前的 `src/governance/` 被称为"治理层"（Governance Layer）。但三框架分析揭示了一个命名冲突：这套机制在本质上是 **Aida 的自我管理工具**，不是外部对 Aida 的治理。

| 维度 | 当前"治理层"实际做的事 | 真正的治理应做的事 |
|------|----------------------|-------------------|
| **执行者** | Aida 自身（ActionGate 在 Aida 进程内运行） | Aida 外部的独立实体 |
| **规则来源** | Aida 自己加载 governance.yaml | 外部权威设定并注入 |
| **监管对象** | Aida 的 5 个写操作工具 | Aida 的全部行为（含文件 I/O、网络、对外通信） |
| **问责方向** | Aida 向自己报告违规 | Aida 向外部委托人报告 |
| **多实例协调** | 不存在 | 协调多个 Aida 实例的权限/资源/冲突 |

### 三框架验证

三个分析框架独立确认了这一区分：

| 框架 | 内部层 = 管理 | 外部层 = 治理 |
|------|-------------|-------------|
| **OS** | Ring 0 内核自保护（MMU, capability check）——内核自己的资源管理 | Ring -1 Hypervisor——审计内核行为、管理硬件资源、隔离多个内核实例 |
| **控制论** | 自调节回路（恒温器根据设定温度自动调节）——控制器即被控对象的一部分 | 外部监督控制（工程师设定恒温器参数、监控恒温器是否正常工作、在恒温器故障时接管） |
| **有限理性** | 自我施加的决策约束（"我的预算规则"）——主体自律 | 制度设计（机构为决策者设计约束环境）——委托人设计代理人的行为边界 |

### 架构影响

**这不仅是命名——它改变推理方向。**

如果内部层叫"治理"：架构给人"治理已完成"的错觉。这解释了为什么"治理绕过"连续 4+ 轮复现却未被视为根本性问题——在"治理"框架下，绕过是"漏洞修补"（P1）；在"管理 vs 治理"框架下，绕过是**自律系统的本质局限**——自律永远有盲区，因为绕过者和执法者是同一个主体。

如果内部层叫"管理"：立即暴露架构的真正缺口——**AIDA 有管理但没有治理**。外部治理层成为一个显式的待建维度。

具体影响：

1. **"治理绕过"重新定性**：Agent 用 `write` 工具绕过 ActionGate 不是"治理漏洞"，而是"管理层内部一致性问题"。真正封堵绕过需要外部治理——从 Aida 的进程**外部**观察其全部行为
2. **路线图重新定位**：P0-P3 的全部改造项都是**管理层增强**，不是"治理改进"。真正的治理是另一个维度
3. **多 Aida 实例场景**：当组织运行多个 Aida 实例时，"谁协调它们"不是管理问题——没有哪个 Aida 应该管理其他 Aida。这需要外部治理层

### 公司治理理论精确对应

| 公司治理术语 | AIDA 中的对应 |
|-------------|-------------|
| **Management**（管理层）= CEO 及管理团队在授权范围内自主运营 | Aida 的内部约束执行（ActionGate, 熔断器, 审批工作流）|
| **Governance**（治理层）= 董事会/股东对管理层的监督、问责、权力边界设定 | OpenClaw 实例外部的监管层（待建） |
| 管理层执行公司政策 | Aida 执行 governance.yaml 中的约束 |
| 管理层内部审计 | Aida 记录违规并在 Dashboard 展示 |
| 管理层风控 | 熔断器保护 |
| 董事会设定 CEO 权限 | 外部治理层定义 Aida 可用的工具集和资源配额 |
| 外部审计师审计财报 | 外部治理层审计 Aida 的全部行为日志 |
| 集团治理协调子公司 | 外部治理层协调多个 Aida 实例 |

### 命名方案

**确立双层命名原则**：

```
外部治理层（Governance Layer）          ← 待建，OpenClaw 实例外部
  观察审计 Aida 行为
  管理权限/规则约定
  维护基础设施
  协调多 Aida 实例
  ───────────────────── OpenClaw 实例边界 ─────────────────────
内部管理层（Management Layer）          ← 当前 src/governance/
  ActionGate 约束执行
  熔断器自保护
  审批工作流
  违规记录
```

**代码命名过渡策略**：
- **现有代码不做机械重命名**——`src/governance/`、`GovernanceStore`、`bps_governance_status` 等保持不变。代码级重命名的工程成本高（4 源文件、2 工具名、1 配置文件、大量测试和文档），但收益边际低
- **新代码和新文档使用新术语**——后续新增的内部约束机制使用 `management` 命名空间；未来外部监管机制使用 `governance` 命名空间
- **CLAUDE.md 和架构文档**中明确标注：`src/governance/` = Management Layer（历史命名）

### 对路线图的影响

P0-P3 全部改造项重新定位为**管理层增强**，新增 P-future 条目：

| 阶段 | 改造项 | 层级 |
|------|--------|------|
| P0-a | 管理层绕过封堵 | Management（内部一致性修复） |

| P1-b | 信息摘要层 | Management（决策辅助） |

| P2-a | 进程组 | Management（批量操作） |

**P-future 外部治理层的初步范围**（概念阶段，不在当前路线图执行范围内）：

### 为什么三框架比"市场即计算"更适合 AIDA

| 维度 | 市场框架 | 三框架 |
|------|---------|--------|
| **匹配度** | 解决去中心化协调 | 解决资源管理(OS) + 稳定性(CTL) + 认知辅助(BR) |
| **可操作性** | 提出"价格信号"等开放性概念 | 提出"deadline 字段"等可直接编码的方案 |
| **可验证性** | "Agent 是否做出经济理性决策"难以测量 | "scan_work 是否返回逾期任务"可直接测试 |
| **风险** | 可能引入不必要的复杂度（货币、竞争） | 每项改动都是 AIDA 已有架构的自然延伸 |

### 三框架间的互补关系

```
                OS 理论
              ↗ （资源管理基础）
AIDA 差距  → 控制论           → 统一优先级排序
              ↘ （稳定性保障）
                有限理性
              （决策环境优化）
```

- **OS 提供骨架**：进程、内存、调度、保护——AIDA 做什么
- **控制论提供心脏**：反馈回路、稳定性、恢复——AIDA 如何自我修正
- **有限理性提供大脑**：信息过滤、决策辅助、满意化——AIDA 如何帮助 Agent 更好地思考

三者正交，各自覆盖 AIDA 的不同维度，没有市场框架中"计算基元"混淆不同抽象层次的问题。
