## 一、项目轨迹概览

AIDA 项目经历了三个阶段性转折，理解这条轨迹比罗列 Phase 更重要：

```
阶段 A（Phase 1-7）：构建
  从零搭建 BPS 理论 → TypeScript 引擎 → 业务蓝图验证 → OpenClaw 集成
  信念：BPS 是 AI-Native 组织运营的计算基础

阶段 B（Phase 8-13）：扩展
  Agent 层（Aida/BPS-Expert/Org-Architect）+ BKM 知识管理 + 项目目录 + Dashboard 部署
  修正：BPS 是结晶机制，不是操作系统（ADR-2）

阶段 C（2026-03-03）：质疑
  BPS 引擎价值反思 — Agent 能力进化正在吞噬 BPS 的运行时价值
  待决：BPS 是否应从"运行时引擎"退化为"描述规范 + 轻量工具"
```

这条轨迹的内在逻辑：每一次修正都在**缩小 BPS 引擎的角色**——从"所有流程必经的管道"→"按需激活的结晶机制"→"可能连运行时引擎都过重"。这不是摇摆，而是随着对 Agent 能力边界的认知加深，BPS 的定位在逐步校准。

## 三、架构演变的三次重大判断

### 判断 1：三层架构分离（Phase 3，2026-02 下旬）

```
BPS 业务流程层（What）→ 智能编排层（When/Who/How many）→ Agent 执行层（How）
```

**评价**：这是项目最重要的架构决策，目前仍然成立。分离避免了过度形式化和失控两个极端。

### 判断 2：BPS 是结晶机制，不是操作系统（Phase 9，2026-03-01）

从"所有流程必经的管道"重新定位为"按需激活的结构化工具"。七个结晶化场景提供判断框架。

**评价**：正确方向，但未走够。接下来的反思证明了这一点。

### 判断 3：BPS 运行时引擎价值重估（2026-03-03）

四个根本性问题的审视：

1. **Agent 能力正在吞噬 BPS 的运行时价值** — Agent 本身就是"活的规则引擎"，7-state 状态机、事件→指令映射、非确定性判断都是 Agent 的本职工作
2. **BPS 理论不应占用 Agent context window** — 500+ 行 BPS 说明书塞进 SOUL/AGENTS 是资源浪费，BPS 应该是 tools 注册，不是 Agent 教育
3. **Skill 是比 BPS Blueprint 更自然的流程固化手段** — Agent 天然理解代码/配置，不需要学习专用 DSL
4. **核心资产是思想框架，不是引擎实现** — "组织运营即图灵计算"的洞见独立于引擎存在

**提出的架构方向**：

```
当前：Agent → BPS Engine (full VM) → 执行业务流程
建议：Agent → Skill/Code (直接执行) + 轻量审计层 (process tracker)
              ↓
            BPS 规范作为"设计时参考"而非"运行时引擎"
```

**状态**：方向已确认，具体瘦身实施尚未开始。

## 六、经验教训

### 架构认知

**1. 对 Agent 能力的低估导致了过度工程化**

项目启动时（2026-02-23），BPS 引擎的设计假设是"Agent 需要一个结构化的运行时框架来约束和编排"。但 Agent 能力的进化速度比预期更快——它本身就能处理状态管理、事件响应、非确定性判断。这导致我们为 BPS 构建了一个完整虚拟机（209 tests），而其中大部分能力 Agent 已经天然具备。

**教训**：在 AI-Native 系统中，应该先验证 Agent 原生能力的边界，再决定需要多少额外框架。

**2. "结晶机制而非操作系统"的判断正确但不够彻底**

ADR-2 将 BPS 从"必经管道"重新定位为"按需激活"是一次重要修正。但修正后仍然保留了完整的运行时引擎，只是在引擎之上加了一层"何时使用"的元判断（结晶化框架）。2026-03-03 的反思进一步指出：**一个工具需要一个决策框架来决定何时使用它，说明工具本身的抽象层次有问题。**

**教训**：架构修正要一步到位。如果发现方向需要调整，应该追问"这个调整是否走得够远"，而不是止步于第一个看起来合理的中间状态。

**3. 理论价值与工程价值需要分离评估**

"组织运营即图灵计算"是一个有深度的理论洞见，BPS 六元组作为描述规范有学术价值。但将这个理论直接实现为运行时引擎，混淆了"理论的正确性"和"实现的必要性"。理论正确不等于需要一个运行时引擎——ISA 定义了指令集，但不意味着每个应用都需要自己写一个 CPU 模拟器。

**教训**：区分"这个想法对不对"和"这个想法需不需要工程化实现"。

### 工程实践

**4. 测试覆盖是重构的安全网，但不能成为保留代码的理由**

209 + 78 = 287 个测试在多次重构中提供了信心（双轨制消除、项目目录迁移、系统蓝图引入）。但"这个模块有 N 个测试"不应成为"这个模块应该保留"的论据。沉没成本不应影响架构决策。

**5. `node:sqlite` 零依赖的选择极大简化了部署**

Node.js 24 内置 SQLite，免去原生编译依赖，任何环境即装即用。这是技术选型中最务实的决策之一。

**6. 代码/数据分离（~/.aida/）越早做越好**

Phase 11 的迁移使得多环境部署和测试独立化成为可能。如果一开始就做，后面很多问题不会出现。

**7. 一键部署脚本的价值在实战中不断被证明**

install-aida.sh 从早期的手动操作演进到 6 步全自动脚本，每次部署问题都推动了脚本的完善。

### 部署与集成

**8. OpenClaw 破坏性升级（2026.3.2）暴露了外部依赖风险**

插件 register 从 async 变 sync、mainAgent 键移除、agents.list 路由语义变化——导致 11 tools **静默丢失**，排查耗费大量时间。关键基础设施的版本升级需要充分的回归测试。

**9. 子 Agent 模型选择是 Agent 系统的隐性瓶颈**

MiniMax-M2.5 作为子 Agent 无法可靠执行工具调用（单轮即停），切换 Gemini 后改善。Agent 系统的可靠性不只取决于框架设计，还高度依赖底层模型的工具调用能力。

**10. CRLF 是 Windows→Linux 部署的必踩之坑**

`.gitattributes` 设 `*.sh text eol=lf` 是标配，且 git clone 比 scp 更安全。

### 流程纪律

**11. 重大决策必须同步更新 CLAUDE.md**

2026-03-03 的架构反思虽然写了完整文档并存入 archive/，但没有更新 CLAUDE.md、session_state.md、ADR 任何一个"活文档"。导致后续会话完全不知道这个决策的存在。

**教训**：已在 CLAUDE.md 顶部增加"会话纪律"——**如果一个决策值得写文档，就值得在 CLAUDE.md 留一行摘要 + 链接。** CLAUDE.md 是每次会话的唯一保证入口，不在这里的信息等于不存在。

## 七、当前状态判断

项目处于一个**架构分叉点**：

- **已建成**：从理论到引擎到 Agent 到 Dashboard 到部署的完整技术栈（287 tests）
- **已质疑**：这个技术栈中 BPS 运行时引擎的必要性
- **未决定**：是继续在完整引擎上推进，还是先执行架构瘦身再推进

这个分叉点的本质是：**AIDA 的核心资产是"AI-Native 组织运营"这个思想框架和 Agent 协作模式，还是 BPS 引擎本身？** 如果答案是前者（目前的判断倾向于此），那么引擎应该服务于思想框架的验证和落地，而不是成为需要维护的独立复杂系统。

**下一步的关键动作**：在真实业务场景中验证——Agent 原生能力 + 轻量工具能覆盖多少需求，哪些需求确实需要 BPS 引擎级别的形式化支持。用事实而非理论来决定 BPS 引擎的最终形态。
