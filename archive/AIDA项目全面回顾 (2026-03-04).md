## 零、根本原则与偏差分析

### AIDA 的根本原则

> **AI-Native 组织应该尽量由 AI 来主导实现人的意图，AIDA 是辅助实现这一目标的基础设施工具。**

这个原则包含三个层次：

1. **AI 主导**：AI Agent 是决策者和执行者，不是框架的"被编排对象"
2. **人的意图**：人提供方向和约束，不提供操作步骤
3. **基础设施**：AIDA 提供持久化、可观测性、审计——而非编排逻辑

### 用根本原则审视项目轨迹

项目的三个阶段，从根本原则视角看，呈现出一条从"违背原则"到"回归原则"的矫正轨迹：

| 阶段 | 做了什么 | 与根本原则的偏差 |
|------|---------|----------------|
| A（Phase 1-7）| 构建 BPS 完整虚拟机：ProcessManager + 7-state 状态机 + 规则引擎 + 9 种 SysCall | **严重偏离**——把 AI 当作框架的执行器，由 BPS 引擎编排 Agent 行为。这等于让基础设施"指挥"AI，而非让 AI 主导 |
| B（Phase 8-13）| 3 个独立 Agent + 500+ 行 BPS 理论塞入 workspace | **中度偏离**——多 Agent 协作拓扑增加了协调开销，BPS 理论占用 context window 降低了 AI 的推理能力。Agent 在被"教育"如何使用框架，而非直接实现意图 |
| C（03-03 反思 + 03-04 重写）| 引擎瘦身、子 Agent 吸收为 Skill、workspace 精简 | **回归中**——承认 Agent 天然具备编排能力，AIDA 开始退回"基础设施"角色 |

### 六项具体偏差

**偏差 1：BPS 引擎试图"编排"AI，而非"服务"AI**

BPS 的运行时引擎（ProcessManager、规则引擎、SysCall）本质上是一个**流程编排器**。它假设 Agent 是"需要被指挥的执行者"——接收 SysCall 指令，按规则触发动作。这与"AI 主导实现意图"完全相悖。

**事实验证**：引擎从未被真实 Agent 调用过（见 2.2 节）。Agent 天然能处理事件响应、状态判断、流程推进——这正是"AI 主导"的体现。

**矫正状态**：engine/ 模块已从 5 文件（ProcessManager + 规则引擎 + SysCall）瘦身为 2 文件（ProcessTracker + StateMachine）。编排逻辑被移除，保留的是追踪和记录——回归基础设施角色。

**偏差 2：多 Agent 协作增加了"人工管理 AI"的负担**

BPS-Expert 和 Org-Architect 作为独立 Agent，需要：子 Agent 模型选择（MiniMax 不可靠 → 切 Gemini）、委派协议设计、响应验证逻辑、红线约束规则。这些"管理 AI 的框架"恰恰违反了"让 AI 主导"——为了让多个 AI 协作，人需要投入大量精力设计协调机制。

**事实验证**：Org-Architect 从未运行过。BPS-Expert 的可靠性问题（模型切换、响应验证）消耗了多个 Phase 的工程精力。

**矫正状态**：两者均已归档，能力提取为 Aida 的 5 个 Skill。Aida 成为唯一 Agent，自给自足——消除了多 Agent 协调的复杂性。

**偏差 3：BPS 理论占用 AI 的认知资源**

将 500+ 行 BPS 理论（六元组、9 种 SysCall、7 种状态、结晶化框架）塞入 Agent workspace，等于用"教材"占据 AI 的工作记忆。AI 需要理解一套专用术语才能开始工作——这是在"教育 AI 使用工具"，而非"让 AI 用自然方式实现意图"。

**矫正状态**：Aida workspace 已重写为英文 30 行 SOUL + 69 行 AGENTS，删除全部 BPS 理论讲解。BPS 知识通过 Skill 按需加载，不再常驻 context window。

**偏差 4：Dashboard 的价值被低估和模糊化**

Dashboard 在回顾报告中被简化为"Layer 1-5"的开发阶段描述，实际它提供了 9 个功能页面和 20+ 个 API。作为"人观测 AI 行为"的主要通道，Dashboard 恰恰是根本原则中最正确的部分——它不干预 AI 决策，只提供可见性。初版报告对其功能的严重低估，反映了当时关注焦点在引擎而非基础设施。

**矫正**：本修订版完整记录 Dashboard 的 9 个页面和实际能力（见 2.3 节）。

**偏差 5：Dossier 和 ~/.aida/ 是最符合原则的设计**

版本化文档存储（Dossier）和项目数据分离（~/.aida/）是纯粹的基础设施——提供持久化，不干预 AI 行为。这些从一开始就没有偏离根本原则。

**偏差 6：BKM 的 5 层知识分级是为框架服务，而非为 AI 服务**

5 层 × 6 域 = 30 种组合的知识分级体系，其复杂度服务于 BPS 引擎的上下文装配需求，而非 AI 的实际知识检索方式。AI 更自然地通过语义搜索和上下文窗口获取知识，不需要预定义的层次结构。

**矫正状态**：knowledge/ 模块已从 5 文件缩减为 3 文件（移除了 context-assembler 和 conflict-detector）。

## 一、项目轨迹概览

AIDA 项目经历了四个阶段性转折：

```
阶段 A（Phase 1-7）：构建
  从零搭建 BPS 理论 → TypeScript 引擎 → 业务蓝图验证 → OpenClaw 集成
  信念：BPS 是 AI-Native 组织运营的计算基础
  ← 偏离根本原则：引擎编排 Agent，而非 Agent 主导

阶段 B（Phase 8-13）：扩展
  Agent 层（Aida/BPS-Expert/Org-Architect）+ BKM 知识管理 + 项目目录 + Dashboard 部署
  修正：BPS 是结晶机制，不是操作系统（ADR-2）
  ← 修正不彻底：多 Agent + 重 context 仍在约束 AI

阶段 C（2026-03-03）：质疑
  BPS 引擎价值反思 — Agent 能力进化正在吞噬 BPS 的运行时价值
  ← 方向确认但未行动

阶段 D（2026-03-04）：矫正
  引擎瘦身已部分落地 + Aida workspace 重写 + 子 Agent 吸收为 Skill
  ← 回归根本原则：AI 主导，AIDA 做基础设施
```

内在逻辑：每一次修正都在**缩小 AIDA 对 AI 行为的干预程度**——从"编排 Agent 的虚拟机"→"按需激活的结晶机制"→"描述规范 + 轻量工具"→"AI 自给自足 + 基础设施服务"。这条轨迹的终点指向根本原则的完全回归：**AIDA 是 AI 的工具，不是 AI 的管理者。**

## 三、架构演变的四次重大判断

### 判断 1：三层架构分离（Phase 3，2026-02 下旬）

```
BPS 业务流程层（What）→ 智能编排层（When/Who/How many）→ Agent 执行层（How）
```

**评价**：基本架构框架仍然成立。但"智能编排层"的实现从"BPS 引擎编排"演进为"AI Agent 自主编排"——编排权已从框架转移到 AI。

### 判断 2：BPS 是结晶机制，不是操作系统（Phase 9，2026-03-01）

从"所有流程必经的管道"重新定位为"按需激活的结构化工具"。七个结晶化场景提供判断框架。

**评价**：正确方向，但未走够。一个工具需要一个决策框架来决定何时使用它，说明工具本身的抽象层次有问题。

### 判断 3：BPS 运行时引擎价值重估（2026-03-03）

四个根本性问题的审视：

1. **Agent 能力正在吞噬 BPS 的运行时价值** — Agent 本身就是"活的规则引擎"
2. **BPS 理论不应占用 Agent context window** — BPS 应该是 tools 注册，不是 Agent 教育
3. **Skill 是比 BPS Blueprint 更自然的流程固化手段** — Agent 天然理解代码/配置
4. **核心资产是思想框架，不是引擎实现** — "组织运营即图灵计算"的洞见独立于引擎存在

## 六、经验教训

### 架构认知

**1. 对 Agent 能力的低估导致了过度工程化**

项目启动时（2026-02-23），BPS 引擎的设计假设是"Agent 需要一个结构化的运行时框架来约束和编排"。但 Agent 能力的进化速度比预期更快——它本身就能处理状态管理、事件响应、非确定性判断。**从根本原则看**，这个错误的根源在于：把 AIDA 定位为"AI 的管理者"而非"AI 的基础设施"。

**教训**：在 AI-Native 系统中，默认假设 AI 能做一切，只在被证明不能时才构建辅助框架。

**2. "结晶机制而非操作系统"的判断正确但不够彻底**

ADR-2 将 BPS 从"必经管道"重新定位为"按需激活"。但修正后仍然保留了完整的运行时引擎，只是在引擎之上加了一层"何时使用"的元判断。

**教训**：架构修正要一步到位。如果发现方向需要调整，应该追问"这个调整是否走得够远"。

**3. 理论价值与工程价值需要分离评估**

"组织运营即图灵计算"是一个有深度的理论洞见。但将这个理论直接实现为运行时引擎，混淆了"理论的正确性"和"实现的必要性"。

**教训**：区分"这个想法对不对"和"这个想法需不需要工程化实现"。

**4. 多 Agent 架构的协调成本被严重低估**

BPS-Expert 和 Org-Architect 的引入不仅没有分散 Aida 的负担，反而创造了新的问题：模型选择（MiniMax → Gemini）、委派协议、响应验证、红线约束。管理 AI 之间的协作，比让一个 AI 直接工作更难。

**教训**：在 Agent 系统中，"一个能力更强的 Agent"几乎总是优于"多个需要协调的 Agent"。复杂性应在 Agent 内部通过 Skill 管理，而非在 Agent 之间通过协议管理。

**5. 文档审查必须基于源代码，而非历史记忆**

本回顾报告初版基于不完整的项目信息生成：引擎实际上已经瘦身（engine/ 5→2 文件），Dashboard 实际上有 9 个功能页面（不只是"Layer 1-5"），Agent 层实际上已经重写——但初版报告对此一无所知。

**教训**：项目回顾必须以`源代码审查`为基础，而非依赖文档记忆或会话上下文。代码是唯一的事实来源。

### 工程实践

**6. 测试覆盖是重构的安全网，但不能成为保留代码的理由**

测试随引擎瘦身从 209 减少到 195，证明了"删代码也要删测试"。沉没成本不应影响架构决策。

**7. `node:sqlite` 零依赖的选择极大简化了部署**

Node.js 24 内置 SQLite，免去原生编译依赖。

**8. 代码/数据分离（~/.aida/）越早做越好**

Phase 11 的迁移使得多环境部署和测试独立化成为可能。

**9. 一键部署脚本的价值在实战中不断被证明**

install-aida.sh 已适配 4 次重大架构变更（OpenClaw 兼容、Dashboard 部署、子 Agent 归档、Skills 部署），每次都证明了自动化的价值。

### 部署与集成

**10. OpenClaw 破坏性升级（2026.3.2）暴露了外部依赖风险**

插件 register 从 async 变 sync、mainAgent 键移除——导致 tools 静默丢失。已通过 install-aida.sh 中的清理逻辑修复。

**11. CRLF 是 Windows→Linux 部署的必踩之坑**

`.gitattributes` 设 `*.sh text eol=lf` 是标配。

### 流程纪律

**12. 重大决策必须同步更新 CLAUDE.md**

2026-03-03 的架构反思没有更新"活文档"，导致后续会话不知道决策存在。已建立会话纪律。

**13. 全面回顾报告必须与源代码交叉验证**

本报告初版的事实偏差（模块文件数、测试计数、Agent 状态、Dashboard 功能范围）证明：基于历史记忆写的回顾报告是不可靠的。项目回顾的正确做法是先审查源代码，再写报告。
