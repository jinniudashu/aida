# 市场即计算：Agent 系统不应建模"快照" (2026-03-11)

> 一次关于 Agent 编排本质的讨论，起于一条推文，连接 AIDA 项目的架构演化实证。

## 起因：推文

> 我一直认为：研究多 Agent 团队结构是邪路。研究记忆、知识结构、上下文管理，和工作流就行了。非要说的话加个异步并发。大模型本来就只有一个，上下文窗口和文件记忆都是流动的。生搬人类的组织结构就只会学到中层病。要透过现象看本质。Agent 编排更像操作系统，而不是人类组织结构。

## 核心类比：市场作为计算机

- **市场** = 计算机内的状态空间
- **市场机制** = 执行逻辑，其中价格信号是核心的资源调度/进程编排机制
- **企业组织** = 被动态生成的计算节点集群，完成特定进程，向价值目标状态迁移
- **人类组织结构** = 这个动态过程的时间片"快照"

**关键命题**：做 Agent 系统不能以"快照"为目标，而应该以**状态迁移函数本身**为目标——不建模"组织长什么样"，而建模"计算如何发生"：调度规则、约束边界、状态转换逻辑、记忆模型。

## 推文说对了什么

推文的核心洞察完全正确：**Agent 编排是 OS，不是 org chart。**

人类组织结构是受限于通信带宽、信任建立成本、注意力瓶颈等物理约束后的妥协方案。Agent 没有这些约束，照搬只会继承病灶而非价值。

"中层病"的本质是**信息路由的低效固化**——中层存在是因为人类无法全局通信，需要层级聚合/分发信息。Agent 的上下文窗口天然就是全局的。

## 推文漏掉的关键维度

推文说"研究记忆、知识、上下文、工作流就行了"——这些都是**计算资源**（内存、存储、指令流）。但 OS 不只有资源管理，还有一个东西：**权限与保护机制**。

用市场类比：市场不只有价格信号调度资源，还有**法律/合规/监管**——定义"不可计算"的边界。没有这层约束，计算节点（企业）会进入非法状态（欺诈、垄断、外部性失控）。

这恰恰是 AIDA 最痛苦也最有价值的发现。

## AIDA 的实证轨迹

AIDA 走过了完整的认知弧线：

**Phase 1-13（快照思维）**：试图用 BPS 建模组织结构——Service 定义、Role 分配、Process 状态机、Rule 编排。本质上在硬编码"组织应该长什么样"。

**架构反思（2026-03-03）**：发现 BPS 大部分运行时价值正在被 Agent 能力进化吞噬。Agent 自己就能决定做什么、怎么做、何时做——你不需要告诉它流程。

**Phase E1 转折点**：Blueprint 从"流程编排器"重定位为"治理宪法"——**定义 Agent 不能做什么，而非应该做什么。**

用市场计算机框架翻译：
- ~~Blueprint = 组织结构图~~ → Blueprint = 约束边界（法律/监管）
- Entity + Skill = 动态计算本身（节点自主决策 + 能力结晶）
- Governance Action Gate = 内核态保护环（ring 0 拦截非法 syscall）

## 更深的洞察

**组织结构是涌现的，不是设计的。** 就像你不会在 OS 里硬编码"进程 A 必须先于进程 B 运行"——你定义调度策略和约束，拓扑自然涌现。

推文说"大模型本来就只有一个"——这在当前技术约束下是对的（上下文窗口是瓶颈），但不是终态。更准确的说法是：

> 不是"只需要一个 Agent"，而是"Agent 的数量和拓扑应该是运行时动态决定的，而非设计时固定的。"

AIDA 的 `agent-create` Skill 和 `skill-create` Skill 就是这个思路——Aida 在运行时根据业务需要动态生成新 Agent 和新 Skill。R7 benchmark 中 Claude Opus 创建了 53 个实体和动态 Skill，GPT-5.4 触发了治理熔断器——这些拓扑都是涌现的，没有人预先设计。

## 一句话总结

**Agent 系统的设计目标不是"模拟组织"，而是"提供计算基元（状态、能力、约束）+ 让结构涌现"。** AIDA 花了 13 个 Phase 才从前者走到后者。
