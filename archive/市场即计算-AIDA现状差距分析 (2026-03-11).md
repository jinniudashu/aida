# 市场即计算：AIDA 现状差距分析 (2026-03-11)

> 基于"市场即计算机"框架，系统审视 AIDA 作为"组织运营计算基础设施"缺失了哪些计算维度。

## 分析框架

市场是一台计算机。它的核心机制不是"组织结构"（那只是快照），而是一组**计算基元**：

| 市场机制 | 计算本质 | 问题域 |
|----------|----------|--------|
| 状态空间 | 可寻址内存 | 世界长什么样？ |
| 价格信号 | 资源调度算法 | 注意力分配给谁？ |
| 产权 | 内存保护 | 谁能改什么？ |
| 合同 | 进程间协议 | 承诺与违约 |
| 货币 | 通用度量 | 不同事物如何比较？ |
| 竞争 | 自然选择 | 谁该存活？ |
| 信号/声誉 | 信息聚合 | 谁值得信任？ |
| 时间价值 | 调度优先级 | 先做什么？ |
| 反馈循环 | 学习算法 | 如何变得更好？ |
| 外部性 | 副作用管理 | 行为的溢出影响？ |

AIDA 应该提供的不是"模拟市场"，而是实现这些计算基元的**引擎原语**，让 Agent 在其上构建涌现秩序。

## 一、已有维度评估

### 1. 状态空间（Entity Dossier）— 成熟度 8/10

**已实现**：版本化 JSON 文档、smartMerge 数组追加语义、ACTIVE/ARCHIVED 生命周期、erpsysId 全局寻址、`getRecentChanges()` 时间序列、不可变审计轨迹。

**差距**：
- **无实体间关系图**：Dossier 是孤立文档，没有外键、依赖声明或关联图。"店铺 A 的地址变了，所有关联订单需要更新"需要 Agent 手动发现和执行，引擎不知道这种传播关系。
- **无约束/验证**：Schema 是纯 JSON，无类型强制、基数规则或引用完整性。Entity 的 `currentValues` 可以是任意结构——灵活但脆弱。
- **无时态查询**："实体 X 在 3 月 5 日的状态是什么？"需要遍历全版本历史手动回放，无原生支持。
- **无变更传播**：`smartMerge` 在 dossier 级别触发事件，无字段级粒度。"价格字段变了"和"备注字段变了"在引擎看来是同一事件。

**市场类比差距**：市场的状态空间是**关联的**——一支股票价格变动会波及整个产业链。AIDA 的状态空间是**平面的**——每个实体是独立内存块，无引用关系，无级联更新。

### 2. 产权与保护（Governance Layer）— 成熟度 7/10

**已实现**：声明式约束、expr-eval 安全求值、BLOCK/REQUIRE_APPROVAL/PASS 三级裁决、熔断器状态机（CLOSED→RESTRICTED→DISCONNECTED）、审批工作流、4 表持久化、5 个写操作工具自动拦截。

**差距**：
- **治理绕过未封堵**：Agent 使用 `write` 工具直接写文件（mock-publish/）绕过治理层，R4-R7 基准测试**连续 4 轮复现**。这是"内存保护"的根本漏洞——相当于用户态程序绕过 MMU 直接写物理内存。
- **静态规则，无学习**：约束从 YAML 加载后不会自适应。不会"观察到 10 次安全执行后自动降级审批要求"，也不会"检测到新攻击模式后自动收紧"。
- **无委托/层级**：没有 RBAC，没有"Alice 可以审批实体更新，Bob 只能审批发布"。审批是平坦的——任何人批准都生效。
- **无违规学习**：违规记录仅用于审计，不反馈到约束定义。"同一类违规发生 20 次"不会触发自动策略调整。

**市场类比差距**：市场的监管是**演化的**——判例法、监管反馈循环、分级执法。AIDA 的治理是**刻板的**——写死的规则 + 二值裁决，更像是一部不会修订的宪法。

### 3. 注意力调度（bps_scan_work + bps_next_steps）— 成熟度 4/10

**已实现**：`bps_scan_work` 一次扫描全工作景观（5 类任务）、`bps_next_steps` 从规则表查询下游服务、priority 字段（整数排序）。

**差距**：
- **无成本模型**：所有任务的执行成本是 0。"生成一篇 2000 字深度分析"和"更新一个标签字段"在引擎看来完全一样。Agent 无法做成本-收益分析。
- **无 SLA/截止时间**：没有 dueDate、deadline 或 estimatedDuration。"这个任务应该在 4 小时内完成"无法表达。
- **无紧急度传播**：高优先级任务的前置依赖不会自动提升优先级。
- **Priority 字段从未被写入**：tools.ts 中的 `bps_create_task` 不暴露 priority 参数，所有任务默认 priority=0，排序实际上退化为创建时间排序。
- **无调度策略声明**：FIFO？优先级加权？最早截止时间优先？完全由 LLM 隐式决定，引擎无法保证一致性。

**市场类比差距**：市场的价格信号是**连续的、多维的、竞争性的**。AIDA 的注意力调度是**离散的、一维的（有/无工作）、非竞争性的**。相当于一个只有"买/不买"按钮、没有价格标签的商店。

## 二、缺失维度

### 4. 货币 / 通用度量 — 完全缺失

**问题**：市场最伟大的发明是**价格**——用一个标量将异质物品变为可比较的。AIDA 没有这个。

"写一篇公众号文章"值多少？"创建一个实体"值多少？"触发一次治理审批"的成本是多少？没有答案。Agent 无法做**经济理性决策**——"我应该花 30 分钟写一篇高质量文章，还是花 5 分钟写 6 篇平庸的？"

**缺失后果**：
- 无法衡量 ROI（投入了什么，产出了什么）
- 无法做资源预算（"今天的 token 预算是 100K，怎么分配？"）
- 无法比较方案（"方案 A 成本 X 预期收益 Y" vs "方案 B 成本 X' 预期收益 Y'"）
- Heartbeat 无法智能排序（"先做收益最高的"——但什么是"收益"？）

**引擎应提供的原语**：
- 任务的 `estimatedCost` 和 `estimatedValue` 字段
- 实体的 `businessValue` 标注
- `bps_scan_work` 返回按 value/cost 比排序的工作列表
- 全局预算概念（每个 heartbeat 周期的 token/时间预算上限）

### 5. 反馈循环 / 学习 — 几乎缺失

**问题**：市场通过**盈亏**学习——赚钱的行为被强化，亏钱的行为被淘汰。AIDA 没有这种系统性反馈。

**当前状态**：
- Agent 的学习仅通过 MEMORY.md（非结构化文本，Agent 自己写自己读）
- 治理层记录违规但不反馈到策略调整
- Skill 创建后无成功率/使用频率追踪
- 实体更新后无"这次更新效果如何"的闭环

**缺失后果**：
- 同样的错误可能反复犯（治理层看到 20 次同类违规，但约束不会自动演化）
- 好的 Skill 和坏的 Skill 同等待遇（无淘汰机制）
- Agent 的决策质量不会随时间提升（除非人类手动调整 Workspace）

**引擎应提供的原语**：
- 任务完成后的 `outcome` 评估（success/partial/failed + reason）
- Skill 的调用计数、成功率、平均耗时
- 治理层的"策略效果"指标（约束 X 拦截了多少次？其中多少被审批通过？→ 约束是否过严？）
- `bps_scan_work` 返回历史成功率（"这类任务过去 80% 成功"）

### 6. 时间价值 / 截止时间 — 几乎缺失

**问题**：市场的时间维度极其丰富——期货、期权、折现率、机会成本。AIDA 的时间概念仅限于 cron 触发。

**当前状态**：
- Cron 提供固定频率触发（每天 10:00、每周一 11:00）
- 任务有 createdAt/updatedAt 但无 deadline/dueDate
- Heartbeat 是无状态的——不知道"上次心跳距今多久"或"有多少逾期任务"

**缺失后果**：
- "这个内容必须在今天 18:00 前发布"无法表达为引擎级约束
- 无逾期检测（"这个任务已经 OPEN 了 3 天没人处理"）
- 无时间折现（"这个任务今天做价值 10，明天做价值 5，后天做价值 0"）
- Agent 无法做"先做紧急的再做重要的"vs"先做重要的再做紧急的"的权衡

**引擎应提供的原语**：
- 任务的 `deadline` 字段 + `bps_scan_work` 按紧急度排序
- 逾期检测事件（`task:overdue`）
- 实体的时间敏感标记（"这条促销信息 3 月 15 日后失效"）

### 7. 竞争与淘汰 — 完全缺失

**问题**：市场通过竞争实现资源的最优配置——多个供应商竞标，低效者被淘汰。AIDA 没有竞争机制。

**当前状态**：
- Skill 一旦创建永不退役
- Agent 一旦创建无性能评估
- 同一任务只有一种执行方式（无 A/B 测试）

**缺失后果**：
- Skill 库会无限膨胀（只增不减），信噪比持续恶化
- 没有"这个 Skill 比那个 Skill 更好"的判断依据
- 无法自动化"淘汰不再使用的能力"

**引擎应提供的原语**：
- Skill 的使用统计（lastUsed、useCount、successRate）
- 自动标记"90 天未使用"的 Skill 为候选归档
- 同类 Skill 的对比（"weekly-report-v1 成功率 60%，v2 成功率 90%"→ 自动推荐 v2）

### 8. 信号与声誉 — 完全缺失

**问题**：市场通过信号（品牌、评级、历史记录）解决信息不对称。AIDA 没有信任/声誉系统。

**当前状态**：
- Agent 对所有实体一视同仁
- 工具调用结果没有可信度标记
- 无"这个数据源可靠吗？"的判断框架

**对多 Agent 场景的影响**：
- 当 Aida 动态创建子 Agent 时，无法评估"这个 Agent 的输出可信吗？"
- 当多个 Skill 提供矛盾建议时，无裁决依据

### 9. 外部性管理 — 完全缺失

**问题**：市场行为有溢出效应（工厂排污影响周边居民）。Agent 操作也有外部性，但无检测机制。

**示例**：
- Agent 大量创建实体 → 数据库膨胀 → 查询变慢（性能外部性）
- Agent 发布内容 → 内容质量影响品牌声誉（业务外部性）
- Agent A 更新实体 → Agent B 的假设失效（信息外部性）

**引擎应提供的原语**：
- 操作的影响范围评估（"这次更新影响了 3 个下游实体"）
- 资源使用配额（"每个 heartbeat 最多创建 10 个实体"）

## 四、架构启示

### AIDA 目前是什么

用市场计算机的语言：AIDA 是一台**有内存（Entity）、有保护机制（Governance）、但没有价格系统、没有调度器、没有学习能力**的计算机。

它更像是一台**早期大型机**——能存储、能执行、有权限控制，但资源分配靠操作员（人类/LLM）手动决定，没有 OS 层的自动化调度。

### AIDA 应该成为什么

一台有**市场机制的计算机**——提供价格信号（注意力定价）、反馈循环（盈亏学习）、时间意识（截止时间 + 折现）、淘汰机制（Skill/Agent 生命周期），让 Agent 组织在其上自发涌现高效结构。

### 最小可行进化路径

不需要同时补齐所有维度。按杠杆率排序：

1. **P0：注意力定价**（成本 + 价值 + 截止时间 → 任务排序）
   - 给 Task 加 `estimatedCost`、`estimatedValue`、`deadline` 字段
   - `bps_scan_work` 返回 ROI 排序的工作列表
   - 这一个改动让 Agent 从"盲目执行"变为"经济理性决策"

2. **P0：反馈闭环**（outcome 评估 → Skill/策略进化）
   - `bps_complete_task` 增加 `outcome` 字段（success/partial/failed）
   - Skill 维度统计（调用次数、成功率）
   - 治理层策略效果报告（拦截/通过比）

3. **P1：治理绕过封堵**（OS 的 ring 0 不能有后门）
   - file write 操作纳入治理检查，或限制 Agent 的文件 I/O 权限
   - 这不是新维度，是已有维度的漏洞修补

4. **P2：Skill 生命周期**（创建 → 使用 → 评估 → 退役）
   - 使用统计 + 自动归档建议
   - 同类 Skill 对比 + 推荐

5. **P3：实体关系图**（从平面内存到关联图）
   - 实体间声明式关系
   - 变更传播语义

## 五、与推文论点的最终对齐

推文说"研究记忆、知识、上下文、工作流就行了"。在市场计算机框架下，这些对应：

| 推文建议 | 市场机制 | AIDA 状态 |
|----------|----------|-----------|
| 记忆 | 状态空间 | ✅ 已有（Entity） |
| 知识 | 信号/信息聚合 | ❌ 空白 |
| 上下文 | — | ✅ OpenClaw 原生 |
| 工作流 | 合同/进程协议 | ⚠️ 部分（Blueprint） |

推文**漏掉的**正是 AIDA **也漏掉的**：价格、反馈、时间、竞争。

这不是巧合——因为这些机制在人类组织中是**隐性的**（老板"凭感觉"分配任务、员工"凭经验"判断优先级、团队"凭直觉"淘汰低效流程），所以在设计 Agent 系统时容易被忽略。但它们恰恰是市场计算机最核心的算法——没有价格信号的市场不是市场，只是仓库。

**AIDA 当前是一个好仓库。下一步是给它装上价格标签。**
