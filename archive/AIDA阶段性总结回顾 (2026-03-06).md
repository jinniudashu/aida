## 二、三次架构转向——项目的思想史

这是整个项目最有价值的部分，记录了认知如何在实践中迭代。

### 转向 1：BPS 从"操作系统"到"结晶机制"（Phase 3, 2026-02）

**初始假设**：所有业务流程必须经过 BPS 建模才能执行。
**发现**：组织运营 80% 是临时性的，只有 20% 需要固化。
**决策**：BPS 是按需激活的工具，不是必经管道。

### 转向 2：BPS 从"运行时引擎"到"基础设施工具"（ADR-10, 2026-03-03）

**初始假设**：Agent 需要框架级的 ProcessManager + RuleEvaluator 来编排行为。
**发现**：Agent 天然具备编排能力——事件判断、任务分解、状态追踪都是 LLM 的本职工作。为 20% 的场景建造了 100% 的虚拟机，投入产出失衡。
**决策**：移除编排层（ProcessManager/RuleEvaluator/SysCall），保留追踪层（ProcessTracker/StateMachine/Dossier）。engine/ 从 5 文件瘦身到 2 文件。

**这是最重要的认知跃迁**：AIDA 的角色从"指挥 AI 做事"变为"给 AI 提供工具和护栏"。

### 转向 3：Blueprint 从"流程编排器"到"治理宪法"（ADR-13, 2026-03-05）

**初始假设**：Blueprint 定义流程拓扑，引擎按拓扑编排 Agent。
**发现**：E2E 测试证明 Aida 完全绕过 Blueprint/Task/Rule，仅用 Entity + Skill 完成了运营管理。
**但也发现**：Agent 越自主，越需要刚性约束——"不是告诉 Agent 做什么，而是确保 Agent 不能做什么"。
**决策**：Blueprint 重定位为治理宪法。新增 Governance 层（ActionGate + CircuitBreaker + governance.yaml）。

## 四、开发节奏模式

回顾 19 个 Phase，有一个清晰的模式：

```
Phase 1-7:   建造虚拟机（ProcessManager + 规则引擎 + SysCall）
Phase 8-13:  赋予 AI 人格和能力（Agent workspace + Skills + Dashboard）
ADR-10:      "虚拟机太重了" -> 拆除编排层，保留追踪层
Phase 14:    "系统封闭" -> 打开对外数据出口
Phase A-D1:  "Agent 不知道自己该做什么" -> 三频模型 + Heartbeat + 自进化
Phase E1-E2: "Agent 太自由了" -> 治理层 + 熔断器 + 审批闭环
03-06 今天:  技术债清理 + P0 修复 + 文档一致性
```

**建造 -> 拆除 -> 重建 -> 约束** —— 这不是走弯路，而是在实践中发现"AI 的基础设施到底应该长什么样"。核心认知从"框架控制 AI"演化为"AI 主导，框架提供工具和护栏"。

## 五、项目当前在什么位置

AIDA 已经完成了**基础设施层**的建设，进入了**业务验证期**。

- **理论层**：BPS 六元组规范 + SBMP 建模方法论 + AGS 治理规范 — 完备
- **引擎层**：34 文件 5,100 行 + 255 tests — 稳定、精简
- **可视化层**：12 页面 + 26 API + SSE — 功能完整
- **Agent 层**：1 Agent + 7 Skills + 13 Tools + 治理护栏 — 可运行
- **文档层**：6 规范 + 13 ADR + 8 归档回顾 — 决策可追溯

下一步的核心问题不再是"基础设施够不够"，而是"在真实业务场景中，这套基础设施好不好用"。
