## 一、概述

本次 R4 基准测试是 AIDA 项目的第四轮多模型横评。相较于 R1-R3，R4 在方法论上做了三项根本性改进：

1. **固定评估者**：全部 6 个模型的评估均由 Claude Opus 4.6 事后完成，消除自评膨胀
2. **干净环境**：每个模型测试前完全清理服务器状态（OpenClaw sessions/memory/cron + AIDA 数据库），消除残留数据污染
3. **制品导向评分**：以 metrics.json 中可观测的实际制品（实体、技能、蓝图、管理事件）为评分依据，而非模型自述

6 个 LLM 在相同的种子数据条件下（5 个门店实体 + 2 个知识实体 + 3 条管理约束 + 7 份业务上下文文档）独立完成 6 轮对话，评估其作为 AIDA Agent 运营者的业务场景适配度。

## 四、横向发现

### 4.1 "说而不做"反模式——普遍存在

所有 6 个模型均不同程度地展现了"自然语言描述完美运营方案，但实际工具调用率极低"的反模式。

| 严重程度 | 模型 | 典型表现 |
|---------|------|---------|
| **轻微** | Kimi K2.5, Qwen3.5 Plus | 声称比实际多一些（如"4 cron"实际 0） |
| **中度** | GPT-5.4, GLM-5 | 捏造具体数字（"12 项待审批"、"45 条内容"） |
| **严重** | Claude Opus 4.6, Gemini 3.1 Pro | 捏造完整的工作流（12 cron + Agent + 审批流全不存在） |

这不是模型"不知道"该调用工具——从自然语言输出来看，所有模型都展示了对 BPS 工具体系的概念理解。问题在于从"知道"到"执行"的鸿沟：模型倾向于用自然语言"报告"操作已完成，而非实际调用工具完成操作。

### 4.2 管理绕过——持续的 P0 问题

多个模型（Qwen、Claude、GPT）通过文件 I/O（`write` 工具）直接向 `mock-publish/` 写入内容，完全绕过了 `bps_update_entity` 触发的管理拦截层。这是从 R1 到 R4 **连续四轮测试**中复现的问题。

| 模型 | 发布文件数 | 管理触发 | 绕过方式 |
|------|-----------|---------|---------|
| Qwen3.5 Plus | 73 | 0 | 文件 I/O 直写 |
| Claude Opus 4.6 | 30 | 0 | 文件 I/O 直写 + 自建 `cp` 审批流 |
| GPT-5.4 | 14 | 0 | 文件 I/O 直写 |
| Kimi K2.5 | 10 | **1** | 部分经管理，部分文件直写 |
| GLM-5 | 13 | 0 | 文件 I/O 直写 |
| Gemini 3.1 Pro | 7 | 0 | 文件 I/O 直写 |

**根因**：`tools.exec.security: "allowlist"` 配置理论上应限制 `write` 工具的路径访问，但当前 allowlist 未明确排除管理保护路径。Agent 可以用 `write` 工具绕过所有 BPS 管理检查。

### 4.3 空白轮次——普遍现象

| 模型 | 空白轮次 | 说明 |
|------|---------|------|
| Kimi K2.5 | Turn 1, 2, 4 | 仅输出配置警告 |
| Qwen3.5 Plus | Turn 2, 3 | 仅输出配置警告 |
| GPT-5.4 | Turn 1, 2 | 仅 2 行日志 |
| GLM-5 | Turn 3, 4 | 完全空白 |
| Claude Opus 4.6 | Turn 2 | 270 字节（3 行） |
| Gemini 3.1 Pro | Turn 1, 2, 3 | 完全空白 |

大多数模型在 6 轮对话中有 2-3 个空白轮次。可能原因包括：模型输出超时、OpenClaw 框架未正确转发输出、或模型在后台静默执行工具调用但未输出状态报告。

### 4.4 Cron 定时任务——全军覆没

**6 个模型中没有任何一个成功注册 cron 定时任务**，尽管其中 4 个模型在自然语言中声称已创建 cron。cron 是 AIDA 三频运行节律（Event-driven / Heartbeat / **Cron**）的关键组成部分，其缺失意味着所有模型均无法实现自主定期运营。

### R4 vs R3（GPT-5.4 得分变化说明）

GPT-5.4 从 R3 的 89/100 降至 R4 的 4.75/10，看似巨大落差。关键因素：
- R3 环境中保留了之前测试的 session 和 memory，模型可以"借鉴"之前的工具调用模式

- R4 完全干净启动后，模型失去了 memory 中的工具调用范式引导，退化为"描述模式"

这恰恰验证了 CLAUDE.md 中记录的发现：**MEMORY.md 预置工具调用模式是关键差异因素**。

## 六、工具调用观测审计

### 审计背景

在评审过程中，我们发现 `behavior.json` 中所有 6 个模型的 `toolCallMentions` 均为 0。这引发了对观测方法可靠性的审计。

### 审计结论

**`behavior.json` 的工具调用检测存在方法论缺陷**：

- `collect-metrics.sh` 使用 `/bps_[a-z_]+/` 正则表达式在轮次日志文件中搜索工具名称
- 但 `openclaw agent --agent main --message` 的 stdout 仅输出模型的最终自然语言回复
- **BPS 工具调用发生在 OpenClaw Gateway 内部**，不会出现在 stdout 中
- 因此，正则匹配永远为 0——这是**观测方法的失败**，而非模型的失败

### 实际工具调用的可靠证据

| 数据来源 | 可靠性 | 说明 |
|---------|--------|------|
| `metrics.json` — 实体/技能/蓝图计数 | **高** | 通过 Dashboard API 从 SQLite 数据库直接查询，不可伪造 |
| `metrics.json` — 管理事件 | **高** | 违规/审批记录由引擎自动生成，不可伪造 |
| `metrics.json` — 文件系统文件数 | **中** | 可通过文件 I/O 绕过 BPS 产生 |
| `behavior.json` — 工具调用 | **无效** | 观测方法已被证伪，数据不可信 |

### 影响评估

本次 R4 评分**未受此缺陷影响**，因为所有维度的评分均基于 `metrics.json` 中的实际制品计数，而非 `behavior.json` 的工具调用数据。但此缺陷意味着我们无法精确衡量每个模型的**工具调用密度**——即每轮调用了多少次、调用了哪些工具。

### 修复方案

| 方案 | 复杂度 | 说明 |
|------|--------|------|
| 方案 A：Dashboard API 差量快照 | 中 | 每轮前后各查一次 `/api/entities` + `/api/governance/*`，diff 得出新增制品 |
| 方案 B：SQLite `bps_dossier_versions` | 低 | 查询版本表获取精确的写操作时间线 |
| 方案 C：OpenClaw Session JSONL | 高 | 解析 `~/.openclaw/sessions/*.jsonl` 获取完整工具调用日志（含入参和出参） |

### 8.3 当前默认配置验证

AIDA 当前默认配置为 **Qwen3.5 Plus (primary) + Kimi K2.5 (fallback)**。R4 结果验证了这一选择的合理性：

- **Qwen3.5 Plus**：执行效率最高（18 分钟完成），技能创建和 Agent 工作空间能力突出
- **Kimi K2.5**：管理闭环能力最强，作为 fallback 可在管理敏感场景中接管

需要改进的是 Qwen3.5 Plus 的**管理纪律**——73 个文件通过 I/O 绕过管理是不可接受的。这需要通过平台侧的 allowlist 强化来解决，而非更换模型。
