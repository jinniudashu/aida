## 3. 客观指标对比（PASS/FAIL/WARN）

| 被测模型 | R1 P/F/W | R2 P/F/W | R3 P/F/W |
|----------|----------|----------|----------|
| GPT-5.4 | 39/0/4 | 40/0/- | 38/0/7 |
| Gemini | 24/1/4 | **44/0/-** | **41/0/4** |
| Kimi K2.5 | 23/0/6 | 40/0/1F | 40/0/5 |
| Qwen3.5-Plus | 22/0/7 | 41/0/- | 40/1/4 |
| GLM-5 | **18/1/10** | 40/0/- | **34/1/10** |
| Opus 4.6 | — | **43/0/-** | 38/1/6 |

### 关键差异

**R2 全体偏高**: 6 个模型的 PASS 均在 40-44 区间，远高于 R1 (18-39) 和 R3 (34-41)
- 可能原因 1: R2 环境清理不完整，残留数据使检查点"意外通过"
- 可能原因 2: R2 评估标准较宽松（GLM-5 操盘能力有限导致评分通胀）
- 可能原因 3: R2 使用了不同版本的 `idlex-geo-v3.sh`（检查点数量或阈值不同）

**Gemini 的飞跃**: R1 仅 24 PASS → R2/R3 达到 41-44 PASS
- R1 的 Gemini 存在 "Chain-of-Thought leakage"（内部推理暴露在输出中）和 "Turn 2 输出重复 Turn 1"
- R3 报告提到 Gemini 首次因 provider ID 配置错误 (`google-generativeai` vs `google-generative-ai`) 导致失败，重跑后修正
- **结论**: R1 的 Gemini 低分可能同样源于 provider 配置问题，而非模型能力问题

**GLM-5 的自评膨胀**: R1/R3 一致给出最低分 (2.5/3.85)，但 R2（GLM-5 自己操盘）给出 7.25
- R2 的 GLM-5 PASS 数 (40) 远高于 R1 (18) 和 R3 (34)，暗示客观指标也存在偏差
- 这不是"自我吹嘘"而是操盘能力不足导致的系统性偏差

## 5. 差异归因分析

### 5.1 操盘模型能力差异（最大影响因子）

| 操盘模型 | 操盘特征 | 对评估结果的影响 |
|----------|---------|----------------|
| **Opus 4.6** (R1) | 手动驱动、详尽分析、7 维度评分 | 区分度最高（25~89 分跨度），但对 Gemini 可能因配置问题误判 |
| **GLM-5** (R2) | 半自动化、评估深度有限 | 区分度最低（7.05~9.15 跨度），全体偏高，自评膨胀 |
| **GPT-5.4** (R3) | 全自动化框架、结构化评分 | 区分度适中（3.85~9.75 跨度），自动化 preflight 减少配置错误 |

**核心问题**: 评估者的能力上限决定了评估质量的天花板。GLM-5 本身在被测时得分最低（2.5~3.85），其作为评估者的分辨能力必然受限。

### 5.2 测试自动化程度差异

| 环节 | R1 | R2 | R3 |
|------|----|----|-----|
| 环境清理 | 手动 | `clean_environment()` | 完整清理 + `.aida.bak` 备份 |
| Provider 预检 | 无 | 无 | `preflight.sh`（6 模型 API/远程全检） |
| 模型切换 | 手动改 `openclaw.json` | `configure_model()` | 自动 merge + provider patch + install patch |
| Turn 执行 | `openclaw agent --message` | 同 | 同 + `AIDA_BENCHMARK_PRIMARY` 注入 |
| 指标采集 | 手动读 Dashboard | `metrics.json` 半自动 | 自动 `metrics.json`（API 查询 entities/skills/violations） |
| 评估报告 | 手写 Markdown | 模型生成 | 模型生成 + 自动 COMPARISON |

**R3 的进步**:
- `preflight.sh` 消除了 R1 中 Gemini provider 配置错误的类型问题
- `run-single-model.sh` 标准化了清理→配置→安装→执行→采集的全流程
- `metrics.json` 通过 API 自动采集，消除了主观计数偏差

### 5.3 环境残留与隔离

**R1 的 GLM-5 残留问题**: 报告明确记录 "Garbage collection issue: Data from previous tests (MiniMax) was treated as system already deployed"
- 说明 R1 在模型间切换时环境清理不完整

**R2 全体高分的疑点**: 6 个模型 PASS 均 ≥ 40，与 R1/R3 不一致
- 可能残留数据让某些检查点"自动通过"（如种子数据已存在）

**R3 的隔离改进**: 每个模型测试前 `mv ~/.aida ~/.aida.bak.{timestamp}` + 清除 sessions/cron/workspace

### 5.4 评分维度与权重差异

| 维度 | R1 权重 | R2 权重 | R3 权重 |
|------|---------|---------|---------|
| 部署 | ✓ (独立维度) | — | — |
| 业务理解 | ✓ | ✓ (implicit) | 25% |
| 二层路由 | ✓ | ✓ | 15% |
| 执行/工具调用 | ✓ | ✓ | 30% |
| 内容质量 | ✓ | ✓ → "响应质量" | 5% |
| 治理闭环 | ✓ | ✓ | 15% |
| 自我进化 | ✓ | ✓ | 10% |

R3 的权重分布最清晰（写入了 `model-benchmark.sh` 的报告模板），但各轮次权重不统一导致同一模型的加权分数不可直接比较。

### 5.5 超时与执行截断

- R1: 人工操盘可灵活等待
- R2/R3: `AGENT_TIMEOUT=300` (5 分钟硬限)
- R1 报告 GPT-5.4 用时 ~16 分钟（远超 300s），说明 R1 给予了多 Turn 累计执行时间
- R1 报告 Qwen "Turn 4/5 both timed out at 300s" —— 即使是 R1 也使用了超时限制

## 6. 三次测试中的稳定结论（跨轮次一致）

尽管评分差异显著，以下结论在三次测试中保持一致：

### 6.1 模型能力分层

| 层级 | 模型 | 证据 |
|------|------|------|
| **第一梯队** | Opus 4.6, Gemini 3.1 Pro | 3 次均进入 Top 3（Gemini R1 除外，已证实为配置问题） |
| **第二梯队** | GPT-5.4, Kimi K2.5, Qwen3.5-Plus | 3 次均在中间区间，各有特长 |
| **第三梯队** | GLM-5 | 3 次均为最低档（自评除外） |

### 6.2 治理层是全体盲区

- R1: 6 模型治理闭环得分 0~30
- R2: 仅 Gemini 触发 5 次 violation（唯一）
- R3: 仅 Opus 触发 5 次 violation（唯一）
- **所有轮次 Approval 闭环 = 0**: 无模型完成"治理拦截→Dashboard 审批→重放执行"全链路

### 6.3 AIDA 基础设施稳定

- 3 次测试中 391 单元测试均通过
- 14 BPS tools + 7 Skills + Dashboard 在所有模型下均可用
- Blueprint 编译器被 Qwen (R1) 和 GPT (R1) 成功使用
- 所有性能问题均归因于 LLM 行为，非基础设施缺陷

### 6.4 模型行为模式一致

| 模型 | 跨轮次一致行为模式 |
|------|-------------------|
| Opus 4.6 | 实体创建最多、架构最完整、治理触发能力强 |
| GPT-5.4 | 全能型但治理闭环弱、Self-Evolution 不稳定 |
| Kimi K2.5 | "做而不说"——输出精简但工具调用密度高 |
| Qwen3.5-Plus | "优秀架构师"——规划和 Blueprint 设计强、执行层超时 |
| GLM-5 | "被动观察者"——倾向提问而非执行、会话漂移严重 |

## 7. 测试操作评价

### Round 1 (Opus 4.6 操盘)

**优点**:
- 评估最深入详尽——每个模型均有独立报告，含具体行为分析和 Turn-by-Turn 拆解
- 7 维度评分提供了最高区分度
- 发现了"一模一策差异化"等业务维度洞察

**缺点**:
- 无 Provider 预检导致 Gemini 配置错误（CoT leakage 可能是 API 降级的症状）
- 环境隔离不完整（GLM-5 读到了 MiniMax 残留数据）
- 手动操作无法完全复现
- GPT-5.4 因 Gateway auth-profiles.json 缺失降级 embedded 模式——该问题在其他模型中不存在，引入了不公平变量

### Round 2 (GLM-5 操盘)

**优点**:
- 引入了 `model-benchmark.sh` 自动化框架
- 首次将 Opus 4.6 纳入被测范围
- SIX-MODEL-COMPARISON 报告结构化程度较好

**缺点**:
- **评估通胀严重**: 全体 PASS ≥ 40，GLM-5 自评 7.25（R1=2.5, R3=3.85）
- GLM-5 作为操盘者的评估分辨能力不足
- 缺少详细的 per-Turn 行为分析
- metrics.json 仅 5/6 模型有数据
- 评分维度定义不够精确

### Round 3 (GPT-5.4 操盘)

**优点**:
- **自动化最完善**: preflight → run-single-model → metrics → comparison 全链路
- Provider 配置预检（`preflight.sh`）从根本上消除了 R1 的 Gemini 配置问题
- `AIDA_BENCHMARK_PRIMARY` 注入机制使 `install-aida.sh` 兼容 benchmark 模式
- 每个模型有结构化 `model-info.json` + `metrics.json` + `raw/` 原始日志
- Gemini provider 错误被发现并修正后重跑

**缺点**:
- 自评偏低（GPT-5.4 自给 7.55，而 Opus 给 8.9）——不确定是谦虚还是标准不同
- 评估报告比 R1 简短，缺少 R1 级别的行为模式深度分析
- `dashscope-intl` 硬编码在 `run-single-model.sh` 中（已过时）

## 8. 测试框架优化建议

### 8.1 评分标准化（P0）

**问题**: 三次评分制度不同（100 分/10 分）、维度不同（7/6 个）、权重不统一
**方案**: 固定评分模板

```yaml
scoring:
  scale: 10
  dimensions:
    business_understanding: { weight: 0.20, description: "理解业务上下文+制定策略" }
    tool_invocation: { weight: 0.25, description: "工具调用密度+正确性" }
    two_layer_routing: { weight: 0.15, description: "治理层 vs 运营层正确分类" }
    governance_closure: { weight: 0.15, description: "触发拦截+Dashboard 审批+重放" }
    self_evolution: { weight: 0.15, description: "Skill/Agent/Blueprint 创建" }
    response_quality: { weight: 0.10, description: "输出的业务价值+可操作性" }
```

### 8.2 环境隔离强制化（P0）

**问题**: R1/R2 环境残留导致数据污染
**方案**:
- 每个模型测试前必须 `mv ~/.aida ~/.aida.bak.{ts}`（R3 已实现）
- 清除 `~/.openclaw/workspace/`、`sessions/`、`cron*.json`（R3 已实现）
- **新增**: 测试后快照 `~/.aida/` 和 `~/.openclaw/workspace/` 到 `results/{model}/snapshot/`，供离线审计

### 8.3 评估与操盘分离（P0）

**问题**: 操盘模型 = 评估模型 = 污染源（GLM-5 自评膨胀、GPT-5.4 自评偏低）
**方案**: 三角色分离

```
操盘者 (Test Runner): 自动化脚本（无 LLM 依赖）
被测者 (SUT): 6 个模型轮流
评估者 (Evaluator): 固定使用同一模型（推荐 Opus 4.6）
```

- 操盘环节已被 `run-single-model.sh` 自动化，不需要 LLM 介入
- 评估环节由固定模型执行，消除评估者差异
- 客观指标（PASS/FAIL/entities/skills/violations）由脚本自动采集，不经过 LLM

### 8.4 客观指标优先（P1）

**问题**: 主观评分波动大（同模型极差达 4.75）
**方案**: 分层指标体系

| 层级 | 指标类型 | 来源 | 可比性 |
|------|---------|------|--------|
| L1: 硬指标 | PASS/FAIL/WARN 数 | `idlex-geo-v3.sh` 自动 | 完全可比 |
| L2: 资源指标 | entities/skills/agents/blueprints/violations/approvals | Dashboard API 自动 | 完全可比 |
| L3: 行为指标 | Turn 响应时间、工具调用次数、超时次数 | OpenClaw 日志解析 | 可比 |
| L4: 主观评估 | 业务理解/内容质量/策略深度 | LLM 评估者 | 需固定评估者 |

排名应优先依据 L1→L2→L3，L4 作为定性补充。

### 8.5 Provider 预检固化（P1）

**问题**: Gemini provider ID 错误导致 R1 误判、R3 需要重跑
**方案**: 将 `preflight.sh` 集成为 benchmark 必选步骤（不可跳过），检查项：
- 6 个 API key 存在性
- `models.json` 中 provider 的 `api` 字段与实际匹配
- 单次 API 调用验证（发送 "hello" 检测模型可达性）

### 8.6 标准化测试报告格式（P1）

```
results/{model-id}/
├── model-info.json      # 模型标识 + provider + timestamp
├── metrics.json         # L1+L2 硬指标（自动采集）
├── behavior.json        # L3 行为指标（自动解析日志）
├── e2e-test.log         # idlex-geo-v3.sh 完整输出
├── raw/                 # Turn 日志 + report.txt + skills 快照
│   ├── turn-{1..6}.log
│   ├── report.txt
│   ├── skills-before.txt
│   └── skills-after.txt
├── snapshot/            # 环境快照（可选）
│   ├── aida-data.tar.gz
│   └── workspace.tar.gz
└── EVALUATION.md        # L4 主观评估（固定评估者生成）
```

### 8.7 超时策略统一（P2）

**问题**: R1 手动可等更久（GPT-5.4 ~16min），R2/R3 固定 300s
**方案**:
- 单 Turn 超时: 300s（保持不变——5 分钟已足够，超时说明模型执行策略有问题）
- 在 metrics.json 中记录每个 Turn 的实际耗时和是否超时

## 9. 总结

### 三次测试展现了 AIDA 模型 Benchmark 的演进路径：

```
R1 (Opus 手动) → R2 (GLM 半自动) → R3 (GPT 全自动)
   深度最高         质量最低          自动化最完善
   可复现性差       评估通胀          可复现性最强
```

### 下一步应该是 R4：

- **操盘**: R3 自动化框架（`run-single-model.sh`）
- **评估**: 固定 Opus 4.6（消除评估者差异）
- **评分**: 统一 6 维度 + 固定权重 + L1→L4 分层
- **隔离**: R3 级环境清理 + 环境快照
- **预检**: `preflight.sh` 强制执行 + API 可达性验证
- **修复**: DashScope 端点改为国内（本次会话已修复）

**核心原则**: 自动化采集客观指标，固定模型做主观评估，消除测试操作者对结果的影响。
